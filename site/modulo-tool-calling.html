Tool Calling e Structured Outputs

Como agentes realmente chamam ferramentas. JSON schemas, parallel tool calls, constrained decoding, JSON mode. A mecânica que separa 'usar a API' de 'construir um agente'.

Seção 16.1

Anatomia de uma chamada com tools

O fluxo canônico, comum a Anthropic, OpenAI e Google:

1. Cliente envia: mensagem do usuário + lista de tools (JSON schema)
2. Modelo responde: texto OU tool_use block (nome + args JSON)
3. Cliente executa a tool e envia tool_result
4. Modelo integra e responde (ou chama mais tools em loop)

Exemplo (Anthropic Messages API):

tools = [{
    "name": "get_weather",
    "description": "Retorna tempo atual",
    "input_schema": {
        "type": "object",
        "properties": {
            "location": {"type": "string"},
            "unit": {"type": "string", "enum": ["celsius", "fahrenheit"]}
        },
        "required": ["location"]
    }
}]

response = client.messages.create(
    model="claude-opus-4-6",
    tools=tools,
    messages=[{"role": "user", "content": "Tempo em Salvador?"}]
)

Seção 16.2

Schemas: a linguagem que os modelos falam

JSON Schema (Draft 2020-12) é o denominador comum. Boas práticas:

  1. Descrições importam mais que nomes. Escreva como doc para humano júnior.
  2. Use enum sempre que puder. Reduz alucinação de valores.
  3. Evite schemas profundos. >3 níveis degrada qualidade.
  4. required não é sugestão. Marque obrigatórios explicitamente.
  5. Exemplos no description. Few-shot dentro do schema melhora.
Seção 16.3

Parallel tool calls

Claude 3.5+, GPT-4+, Gemini 1.5+ retornam múltiplas tool calls na mesma resposta quando são independentes.

Sempre implemente execução paralela no cliente

Use asyncio.gather() ou equivalente. O modelo já fez o trabalho cognitivo de decidir a independência — cobre-lhe a dívida de latência.

Seção 16.5

Structured Outputs: além de tool calling

  • JSON mode (OpenAI response_format: json_object) — garante JSON válido.
  • Strict schema (OpenAI json_schema strict) — conforme schema, 100% garantido.
  • Anthropic: sem modo JSON dedicado; use tool com uma só tool como workaround.
  • Gemini: response_mime_type="application/json" + response_schema.
Seção 16.6

Decodificação restrita (constrained decoding)

Como garantir 100% de conformidade ao schema? Modificando a etapa de amostragem:

A cada token, antes do softmax:
  1. Computar quais tokens são válidos dado o parser state
  2. Zerar probabilidade dos inválidos
  3. Amostrar apenas dos válidos

Frameworks:

  • Outlines (Willard & Louf 2023) — regex, JSON Schema, Pydantic, CFG. github
  • XGrammar (Dong 2024) — estado da arte em performance, integrado ao vLLM.
  • GBNF (llama.cpp) — grammar-based Naur form.
  • LMQL — linguagem de query para LLMs.
  • BAML — schema-first DX focado em TS/Python.
  • Instructor — wrapper Pydantic-first.

Seção 16.7

Padrões de erro e recuperação

  1. Tool name alucinada — validação + mensagem de erro explícita ao modelo.
  2. Argumentos malformados — use structured outputs ou valide + pergunte novamente.
  3. Loop infinito — limite de iterações, timeout total, detecção de repetição.
  4. Tool timeout — retorne tool_result com is_error: true.
  5. Output muito grande — truncar + resumir antes de retornar.
