05

Prompt Engineering

Técnicas fundamentais e avançadas para formular instruções eficazes: Zero-shot, Few-shot, Chain-of-Thought, Role Prompting e padrões específicos para código.

Seção 5.1

O que é Prompt Engineering?

Prompt Engineering é a disciplina de formular instruções e inputs para modelos de linguagem de forma a obter os melhores resultados possíveis. É a ponte entre a intenção humana e a compreensão da máquina.

Enquanto a Engenharia de Contexto decide O QUE enviar ao modelo, Prompt Engineering decide COMO formular o que é enviado.

Seção 5.2

Técnicas Fundamentais de Prompt Engineering

5.2.1 Zero-Shot Prompting

O modelo recebe apenas a instrução, sem exemplos:

PromptPrompt: "Escreva uma função Python que valida endereços de email."

# Útil quando a tarefa é bem definida e o modelo tem conhecimento suficiente

5.2.2 Few-Shot Prompting

O modelo recebe exemplos do formato desejado antes da tarefa:

PromptPrompt: "Converta estas descrições em schemas SQL:

Descrição: 'Tabela de usuários com nome, email e data de criação'
SQL: CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    name VARCHAR(255) NOT NULL,
    email VARCHAR(255) UNIQUE NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

Descrição: 'Tabela de pedidos com valor total e status'
SQL: CREATE TABLE orders (
    id SERIAL PRIMARY KEY,
    total_amount DECIMAL(10,2) NOT NULL,
    status VARCHAR(50) DEFAULT 'pending',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

Descrição: 'Tabela de produtos com nome, preço e categoria'
SQL:"
📚
Referência

Brown, T. et al. (2020). "Language Models are Few-Shot Learners". arXiv:2005.14165. arxiv.org/abs/2005.14165

5.2.3 Chain-of-Thought (CoT)

Instruir o modelo a "pensar passo a passo" antes de dar a resposta final:

PromptPrompt: "Analise este código e identifique o bug.
Pense passo a passo: primeiro entenda o que o código faz,
depois trace a execução com dados de exemplo,
e finalmente identifique onde o comportamento diverge do esperado.

```python
def binary_search(arr, target):
    low, high = 0, len(arr)
    while low <= high:
        mid = (low + high) // 2
        if arr[mid] == target:
            return mid
        elif arr[mid] < target:
            low = mid + 1
        else:
            high = mid + 1  # Bug aqui
    return -1
```"
📚
Referência

Wei, J. et al. (2022). "Chain-of-Thought Prompting Elicits Reasoning in Large Language Models". arXiv:2201.11903.

5.2.4 Role Prompting (Atribuição de Papel)

Definir um papel específico para o modelo:

PromptPrompt: "Você é um engenheiro de software sênior especializado em
segurança de aplicações web. Revise o seguinte código de autenticação
e identifique vulnerabilidades OWASP Top 10. Para cada vulnerabilidade
encontrada, explique o risco e forneça o código corrigido."

5.2.5 Structured Output (Saída Estruturada)

Solicitar resposta em formato específico:

PromptPrompt: "Analise este pull request e responda EXATAMENTE neste formato JSON:
{
  'aprovado': boolean,
  'problemas': [
    {
      'arquivo': 'caminho/do/arquivo',
      'linha': número,
      'severidade': 'critical' | 'warning' | 'suggestion',
      'descrição': 'texto',
      'correção_sugerida': 'código'
    }
  ],
  'resumo': 'texto'
}"
Seção 5.3

Técnicas Avançadas de Prompt Engineering

5.3.1 System Prompts

O system prompt é a instrução "invisível" que define o comportamento base do modelo. Ferramentas como Claude Code usam system prompts extensos:

System PromptSystem: "Você é Claude Code, o assistente de programação da Anthropic.
Você opera no terminal do usuário e tem acesso a ferramentas para
ler, editar e criar arquivos, executar comandos bash, e buscar
no código.

Regras:
1. Sempre leia um arquivo antes de editá-lo
2. Prefira editar arquivos existentes a criar novos
3. Não adicione funcionalidades além do solicitado
4. Não introduza vulnerabilidades de segurança
..."

5.3.2 Prompt Chaining (Encadeamento de Prompts)

Dividir tarefas complexas em etapas sequenciais onde a saída de um prompt alimenta o próximo:

WorkflowEtapa 1: "Analise os requisitos e liste as entidades do sistema"
    → Saída: [User, Order, Product, Payment]

Etapa 2: "Para cada entidade, defina o schema do banco de dados"
    → Saída: SQL schemas

Etapa 3: "Gere os modelos ORM a partir dos schemas"
    → Saída: Código dos modelos

Etapa 4: "Gere os endpoints REST CRUD para cada modelo"
    → Saída: Código das rotas

5.3.3 Self-Consistency

Gerar múltiplas respostas e selecionar a mais consistente:

PromptPrompt: "Gere 3 abordagens diferentes para implementar cache
nesta API. Para cada abordagem, liste prós e contras.
Depois, recomende a melhor opção justificando sua escolha."
📚
Referência

Wang, X. et al. (2022). "Self-Consistency Improves Chain of Thought Reasoning in Language Models". Google Research. arXiv:2203.11171. arxiv.org/abs/2203.11171

5.3.4 Extended Thinking (Pensamento Estendido)

Modelos como Claude oferecem um modo de "pensamento estendido" onde o modelo raciocina internamente antes de responder. Isso é especialmente útil para:

  • Problemas de arquitetura complexos
  • Debugging de bugs sutis
  • Decisões de design com múltiplos trade-offs
📚
Referência

Anthropic. "Extended Thinking". docs.anthropic.com/en/docs/build-with-claude/extended-thinking

Seção 5.4

Anti-Padrões de Prompt Engineering

Anti-Padrão Problema Correção
Prompt vago "Melhore este código" "Refatore para separar a lógica de validação da lógica de negócio"
Prompt contraditório "Seja conciso mas explique em detalhes" Escolha um estilo e seja consistente
Excesso de instruções 50 regras no prompt Priorize as 5-10 regras mais importantes
Falta de formato "Gere testes" "Gere testes usando pytest, com pelo menos um teste para o caminho feliz e um para cada caso de erro"
Antropomorfização "Pense como um humano" "Analise sistematicamente cada condição do if/else"
Seção 5.5

Prompt Engineering Específico para Código

Ao programar com IA, há padrões específicos que funcionam melhor:

1. Forneça o contexto do erro completo

Ruim
"O código não funciona"
Bom
"Ao executar `pytest test_auth.py`, recebo
o seguinte erro:
[cole o traceback completo]
O erro ocorre na linha 45 de auth_service.py"

2. Especifique o framework e versão

Ruim
"Crie uma API REST"
Bom
"Crie um endpoint POST /api/users usando
FastAPI 0.104+, com Pydantic v2 para validação
e SQLAlchemy 2.0 para ORM"

3. Indique as convenções do projeto

Prompt# BOM:
"Este projeto usa:
- Python 3.12, FastAPI, SQLAlchemy 2.0
- Padrão Repository para acesso a dados
- Testes com pytest e fixtures para DB
- Nomes de variáveis em snake_case
- Docstrings no formato Google"

4. Delimite o escopo claramente

Prompt# BOM:
"Adicione validação de CPF à função create_user em
user_service.py. NÃO modifique outros arquivos.
Use a biblioteca validate-docbr já instalada no projeto."
💡
Dica

Quanto mais específico e contextualizado for o seu prompt ao programar com IA, menos iterações serão necessárias para obter o resultado desejado. Um bom prompt de código inclui: o que fazer, onde fazer, como fazer (convenções), e o que não fazer.

Seção 5.6

Alucinações: Como Mitigar na Programação

Como vimos no Módulo 1 — Seção 1.5, alucinações são uma propriedade intrínseca de LLMs: não podem ser eliminadas, mas podem ser drasticamente reduzidas com as técnicas certas. Esta seção é um guia prático para minimizar alucinações ao programar com IA.

💡
Princípio central

A melhor arma contra alucinação é não deixar o modelo adivinhar. Alimente-o com contexto verificável e force verificação automática do resultado. Qualquer decisão que dependa apenas do "conhecimento interno" do modelo é uma aposta.

5.6.1 Estratégias de Grounding (Aterramento)

Grounding significa "ancorar" a resposta do modelo em material concreto fornecido no contexto. Em vez de deixar o modelo recuperar da memória (onde alucinações nascem), você entrega a verdade na bandeja.

  • Contexto concreto: sempre forneça o código real do arquivo em vez de descrever o que ele faz. "Aqui está auth_service.py, linhas 30-80" é muito melhor que "eu tenho um serviço de autenticação que usa JWT".
  • Documentação inline: cole trechos da documentação oficial da biblioteca no prompt. Se você quer que o modelo use httpx.AsyncClient corretamente, cole os 20 linhas da doc dos métodos relevantes antes de fazer a pergunta.
  • Esquemas e tipos: forneça schemas Pydantic, interfaces TypeScript, protobuf, JSON Schema, OpenAPI spec, DDL SQL. Tipos são contratos verificáveis — o modelo fica forçado a respeitá-los.
  • Exemplos de entrada/saída: 2-3 exemplos concretos de I/O fazem mais pelo grounding do que um parágrafo de descrição.
Ruim (sem grounding)
"Crie uma função que chama a API
do Stripe para criar um pagamento"
Bom (com grounding)
"Usando stripe-python 8.x, crie uma função
create_payment(amount_cents: int, currency: str,
customer_id: str) -> stripe.PaymentIntent.

Doc oficial relevante:
[cole o trecho de stripe.PaymentIntent.create]

Nosso schema interno:
[cole o Pydantic model PaymentRequest]"

5.6.2 Verificação Ativa (Verification Loops)

Nunca confie, sempre verifique. A LLM produz; uma ferramenta determinística valida. Esse loop é a diferença entre "IA gerando código" e "IA entregando código funcional".

1. Gerar
LLM produz o código candidato.
2. Compilar / Type-check
tsc, mypy, pyright, cargo check — detecta APIs fantasma e erros sintáticos.
3. Lint
eslint, pylint, ruff — detecta imports fantasmas e padrões suspeitos.
4. Executar testes
pytest, vitest, go test — detecta alucinações semânticas.
5. Feedback → 1
Se falhar, devolva o erro ao modelo para regenerar.
  • Execute o código: não confie, rode. Claude Code faz isso nativamente via ferramenta Bash, executando scripts e capturando stdout/stderr para autocorreção.
  • Rode os testes: configure o agente para rodar a suíte de testes automaticamente após gerar código e iterar até passar.
  • Type checker no loop: mypy --strict, tsc --noEmit, pyright. Esses ferramentas detectam 80% das APIs fantasma antes do código sequer rodar.
  • Linters como guardrails: ruff, eslint, pylint. Catálogos curados de anti-padrões — incluindo imports de módulos inexistentes.

5.6.3 RAG e Busca na Documentação

RAG (Retrieval-Augmented Generation) injeta no contexto do modelo trechos relevantes recuperados de uma fonte de verdade antes da geração. É um grounding automatizado.

  • Documentação em tempo real: forneça trechos específicos da documentação oficial no contexto em vez de confiar na memória do modelo.
  • MCP servers de documentação: ferramentas como Context7 MCP, DeepWiki MCP e Ref MCP buscam documentação atualizada de bibliotecas em tempo real e injetam no contexto do agente (veja Módulo 12).
  • Indexe o seu próprio código: RAG sobre o repositório garante que o modelo use suas próprias utilidades em vez de inventar novas.
  • Cuidado com o training cutoff: qualquer biblioteca que evoluiu após a data de corte do modelo é território fértil para API desatualizada — RAG é obrigatório nesses casos.
⚠️
Training cutoff ≠ conhecimento atualizado

Um modelo treinado até outubro/2024 não sabe que o Next.js 15 lançou unstable_after(), ou que o SQLAlchemy 2.1 mudou o comportamento de Session.merge(). Sem RAG ou fetch de documentação, ele vai inventar com confiança algo baseado em versões antigas.

5.6.4 Extended Thinking e Self-Critique

Modelos modernos (Claude Opus/Sonnet com extended thinking, o1/o3 da OpenAI, Gemini com thinking) podem gastar tokens extras raciocinando antes de responder. Esse "rascunho interno" reduz substancialmente alucinações — o modelo tem oportunidade de checar consistência, questionar hipóteses e considerar alternativas.

  • Ative extended thinking para tarefas críticas (refatoração, bugs sutis, segurança).
  • Peça self-critique explícito: "Revise sua resposta e procure erros", "Liste 3 coisas que podem estar erradas na sua solução".
  • Force verificação de premissas: "Antes de responder, verifique se cada função que você usa existe na API atual da biblioteca. Para cada import, cite de onde vem."
  • Chain-of-verification (CoVe): peça ao modelo (a) gerar a resposta, (b) gerar perguntas de verificação, (c) respondê-las, (d) refinar a resposta.
Sem self-critique
"Escreva uma função que valida
um JWT usando PyJWT."
Com self-critique
"Escreva uma função que valida um JWT
usando PyJWT 2.8+.

Depois de escrever:
1. Liste cada import e confirme que
   existe no PyJWT 2.8+.
2. Liste cada parâmetro de jwt.decode
   e confirme a assinatura atual.
3. Identifique 3 erros comuns que
   sua implementação poderia ter.
4. Corrija se encontrar algum."

5.6.5 Human-in-the-Loop

Nenhuma técnica anterior substitui revisão humana. A IA acelera, mas a responsabilidade pelo código continua sua.

  • Nunca aceite cegamente: nem o código "óbvio" de uma linha. Alucinações semânticas costumam se esconder exatamente no trivial.
  • Code review obrigatório: trate PRs gerados por IA com o mesmo (ou maior) rigor de PRs humanos.
  • Testes de aceitação: valide comportamento real, não só happy path. Teste casos limite que o modelo não pensaria em cobrir.
  • Granularidade: prefira gerar mudanças pequenas e revisáveis a patches gigantes.
  • Pair com a IA, não delegue: use IA como colega de pair programming (você dirige, ela sugere), não como subordinado (você aceita, ela decide).

5.6.6 Técnicas Específicas por Tipo de Alucinação

Prevenção de API fantasma

Ruim
"Use pandas para ler um arquivo
Excel grande com streaming"
Bom
"Usando pandas 2.2, leia um Excel
grande. Use APENAS APIs documentadas
em https://pandas.pydata.org/docs/reference/
— não invente métodos.
Se a funcionalidade de streaming não
existir nativamente, diga isso
explicitamente e proponha alternativa
com openpyxl em modo read_only."

Prevenção de API depreciada

Ruim
"Concatene dois DataFrames"
Bom
"Usando pandas 2.2 (DataFrame.append
foi REMOVIDO na 2.0), concatene
dois DataFrames. Use pd.concat.
Confirme a versão antes de sugerir."

Verificação de correção semântica

Ruim
"Implemente busca binária
em uma lista ordenada"
Bom
"Implemente busca binária em lista
ordenada. Requisitos:
- O(log n) comprovadamente
- Retorne -1 se não encontrado
- Inclua testes com: lista vazia,
  1 elemento, elemento no início/fim/
  meio, elemento ausente
- Rode os testes e mostre a saída"

5.6.7 Ferramentas Anti-Alucinação

Ferramenta Mecanismos anti-alucinação
Claude Code Extended thinking nativo, verification loops via Bash (roda testes, type check, lint), integração MCP para RAG de docs em tempo real, agentes especializados (subagents) para revisão.
GitHub Copilot Sugestões inline grounded no contexto do repositório (symbols, arquivos vizinhos, histórico git). Chat com workspace-aware retrieval reduz APIs fantasma.
Cursor Chamada de ferramentas MCP para verificação, indexação RAG do repositório, modo Agent que executa comandos e lê output para se autocorrigir.
Type checkers mypy, pyright, tsc, flow. Detectam APIs inexistentes estaticamente — seu primeiro filtro anti-alucinação.
Linters ruff, pylint, eslint, biome. Detectam imports suspeitos, uso de APIs depreciadas, padrões de código improváveis.
MCP de docs Context7, DeepWiki, Ref. Injetam doc atualizada no contexto do agente antes da geração — neutraliza training cutoff.
Frameworks de teste pytest, vitest, jest. Execução rápida em loop detecta alucinações semânticas. Property-based testing (hypothesis, fast-check) expõe casos que o modelo não anteviu.

5.6.8 Checklist Anti-Alucinação

✅
Antes de aceitar qualquer código gerado por IA
  1. Versões confirmadas: o prompt especificou as versões exatas das bibliotecas usadas?
  2. Imports verificados: todos os import / require apontam para símbolos que realmente existem nessas versões?
  3. APIs existem: cada função, método ou classe usada está na documentação oficial atual? (sem confiar só na memória do modelo)
  4. Type check passa: mypy, tsc, pyright ou equivalente rodou sem erros?
  5. Lint limpo: ruff, eslint ou equivalente sem warnings relevantes?
  6. Testes rodam e passam: a suíte completa (não só o happy path) foi executada?
  7. Casos limite cobertos: lista vazia, null, valores negativos, unicode, concorrência — o que a LLM poderia ter ignorado?
  8. Segurança revisada: sem SQL injection, sem secrets hardcoded, sem deserialização insegura, sem eval?
  9. Revisão humana explícita: um humano leu, entendeu e aprovou — não apenas "LGTM"?
