21

Otimização de Custo e Eficiência

Reduzir custo em 5–10× sem sacrificar qualidade. Cascading (FrugalGPT), caching semântico, speculative decoding, batch APIs, prompt compression e distillation.

Seção 19.1

Model cascading: small first, escalate

FrugalGPT provou que cascatear modelos pode reduzir custo em até 98% sem perda de qualidade.

1. Classifier pequeno decide complexidade
2. Fácil → Haiku 4.5 / GPT-5-mini ($0.30/1M)
3. Médio → Sonnet 4.6 / GPT-5 ($3-5/1M)
4. Difícil → Opus 4.6 / o3 ($15/1M)
📚
Referências

Chen, L. et al. (2023). "FrugalGPT: How to Use Large Language Models While Reducing Cost and Improving Performance". Stanford, arXiv:2305.05176.

Seção 19.2

Semantic caching

Cache exato (hash do prompt) não serve para queries variadas. Semantic cache compara embeddings — se cos sim > threshold, serve resposta cacheada.

  • GPTCache — github
  • Redis Vector — integrado.
  • Momento — serverless com vetor.
Cuidados
  • Threshold alto (≥0.95) para não servir respostas erradas.
  • Invalidação via TTL + freshness.
  • Cacheie só quando contexto variável não afetar a resposta.
Seção 19.3

Prompt caching provider-side

  • Conteúdo estático primeiro (system prompt, docs, tools).
  • Conteúdo variável por último (mensagem do usuário).
  • Desconto de 50–90% em cache hits (Anthropic, OpenAI, Google).
  • TTL típico: 5 min (estendido para 1h em alguns provedores).
Seção 19.4

Speculative decoding

Modelo pequeno (draft) propõe N tokens; modelo grande (target) valida em paralelo. Ganho: N tokens pelo preço de um forward pass.

Implementado pelos provedores internamente (vLLM, TensorRT-LLM). Saber que existe explica por que latência não é linear com tamanho do modelo.

📚
Referências

Leviathan, Y., Kalman, M. & Matias, Y. (2023). "Fast Inference from Transformers via Speculative Decoding". Google Research, arXiv:2211.17192.

Seção 19.6

Prompt compression

Comprimir prompt mantendo informação. Técnica: modelo pequeno identifica tokens descartáveis e remove. Compressão 2–20× com perda mínima.

📚
Referências

Jiang, H. et al. (2023). "LLMLingua". Microsoft, arXiv:2310.05736.

Jiang, H. et al. (2023). "LongLLMLingua". arXiv:2310.06839.

Seção 19.7

Knowledge distillation

Treinar modelo pequeno (student) para imitar modelo grande (teacher) em tarefa específica.

Padrão 2026: gerar dataset de alta qualidade com Opus/o3, fine-tunar Haiku/GPT-5-mini no dataset. Custo 10–50× menor, qualidade competitiva na tarefa alvo.

📚
Referências

Hinton, G., Vinyals, O. & Dean, J. (2015). "Distilling the Knowledge in a Neural Network". arXiv:1503.02531.

Taori, R. et al. (2023). "Alpaca". Stanford.

Mukherjee, S. et al. (2023). "Orca". arXiv:2306.02707

Seção 19.8

Roteamento entre provedores

  • OpenRouter — openrouter.ai — proxy unificado.
  • LiteLLM — self-hosted, mesma ideia.
  • Martian — roteamento ML-otimizado.
Checklist

Antes de ativar reasoning extendido ou comprar capacity

  1. O prompt está usando prompt caching corretamente?
  2. Conteúdo variável vem no final?
  3. Consigo rotear para modelo menor em 70% dos casos?
  4. Tenho batch workload que pode ir para Batch API?
  5. Uso streaming para melhorar latência percebida?
  6. Cache semântico cobre queries repetitivas?
  7. Output tokens otimizados? (respostas curtas quando possível)
