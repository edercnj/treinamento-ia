Rules e Arquivos de Configuração

Como personalizar o comportamento de ferramentas de IA para seu projeto usando CLAUDE.md, agents.md, .cursorrules e copilot-instructions.md.

Seção 8.1

O Conceito de Rules

Rules são instruções persistentes que personalizam o comportamento de ferramentas de IA para um projeto ou organização específica. Elas funcionam como um manual de estilo vivo que é automaticamente incluído no contexto toda vez que a IA é invocada.

Analogia

Rules são como o .editorconfig ou .eslintrc do mundo da IA — definem as "regras do jogo" para o assistente, assim como linters definem regras para o código.

Por que Rules existem:

  1. Consistência: Todo membro da equipe recebe as mesmas instruções da IA
  2. Contexto automático: Informações críticas sobre o projeto são fornecidas sem que o desenvolvedor precise repeti-las
  3. Personalização: A IA se adapta às convenções, stack e práticas do time
  4. Eficiência: Evita reexplicar as mesmas coisas a cada conversa
Seção 8.2

CLAUDE.md — O "Manual do Projeto" para Claude

O arquivo CLAUDE.md é o mecanismo de rules do Claude Code. Ele é automaticamente lido e incluído no contexto do modelo no início de cada sessão.

Localização e hierarquia:

⚙ ~/.claude/CLAUDE.md Global (todas as máquinas do usuário)
⚙ ~/projeto/CLAUDE.md Raiz do projeto (commitado no git)
⚙ ~/projeto/src/CLAUDE.md Diretório específico (contexto local)
⚙ ~/projeto/.claude/CLAUDE.md Local do projeto (não commitada)
ℹ
Hierarquia de Carregamento

Todos os níveis são carregados e combinados. O mais específico complementa (não substitui) o mais geral.

Estrutura recomendada de um CLAUDE.md:

markdown# CLAUDE.md

## Visão Geral do Projeto
Breve descrição do que o projeto faz e sua arquitetura.

## Stack Técnica
- Linguagem: Python 3.12
- Framework: FastAPI 0.104
- Banco de dados: PostgreSQL 16 + SQLAlchemy 2.0 (async)
- Testes: pytest + pytest-asyncio
- Linter: ruff
- Formatter: black

## Comandos Importantes
- `make test` — roda todos os testes
- `make lint` — roda linter
- `make dev` — inicia servidor de desenvolvimento
- `docker compose up -d` — sobe dependências

## Convenções de Código
- Use snake_case para funções e variáveis
- Use PascalCase para classes
- Docstrings no formato Google
- Imports absolutos (from app.services import ...)
- Repository pattern para acesso a dados
- Cada rota em arquivo separado em app/routers/

## Padrões Arquiteturais
app/
├── routers/      ← Endpoints (thin controllers)
├── schemas/      ← Pydantic models (request/response)
├── services/     ← Lógica de negócio
├── repositories/ ← Acesso a dados
├── models/       ← SQLAlchemy models
└── core/         ← Config, deps, security

## Regras Importantes
- SEMPRE rode `make lint` antes de commitar
- NUNCA modifique arquivos em migrations/ manualmente
- Testes devem usar fixtures do conftest.py para DB
- Variáveis de ambiente vão em .env (NÃO commitar)

## Contexto de Negócio
Este é um sistema de e-commerce B2B. Os "usuários" são empresas,
não pessoas físicas. Pedidos podem ter múltiplos endereços de entrega.

Seção 8.3

agents.md — Instruções para Agentes Específicos

O arquivo agents.md (ou diretório .claude/agents/) permite definir agentes especializados no Claude Code — agentes com instruções, ferramentas e comportamentos customizados para tarefas específicas.

Exemplo de estrutura:

📁 .claude/
📁 agents/
📄 code-reviewer.md Agente de revisão de código
📄 test-writer.md Agente de escrita de testes
📄 db-migration.md Agente de migrações de banco

Exemplo de um agente especializado

Arquivo: .claude/agents/code-reviewer.md

markdown---
name: Code Reviewer
description: Revisa código focando em segurança, performance e boas práticas
tools:
  - Read
  - Grep
  - Glob
  - Bash
---

# Instruções

Você é um revisor de código sênior. Ao receber código para revisão:

1. Leia TODOS os arquivos modificados
2. Verifique cada um dos seguintes critérios:
   - [ ] Segurança (OWASP Top 10)
   - [ ] Performance (queries N+1, loops desnecessários)
   - [ ] Tratamento de erros
   - [ ] Cobertura de testes
   - [ ] Aderência às convenções do CLAUDE.md

3. Produza um relatório no formato:
   ## Resumo
   ## Problemas Críticos
   ## Sugestões de Melhoria
   ## Aprovação: SIM/NÃO

Nunca modifique arquivos — apenas analise e reporte.
Seção 8.4

Templates Completos por Ferramenta

Cada ferramenta de IA tem seu próprio formato de rules, localização de arquivos e mecanismos de ativação. Esta seção apresenta templates prontos, problemas resolvidos e boas práticas específicas para as principais ferramentas do mercado.

8.4.1 Claude Code — CLAUDE.md

O mecanismo nativo de rules do Claude Code. Todos os arquivos encontrados na hierarquia são carregados e combinados — o mais específico complementa (não substitui) o mais geral.

⚙ ~/.claude/CLAUDE.md Global (todas as máquinas do usuário)
⚙ ./CLAUDE.md Raiz do projeto (commitado no git)
⚙ ./.claude/CLAUDE.md Local do projeto (não commitado)
⚙ ./src/CLAUDE.md Contexto específico de subdiretório

Template expandido

CLAUDE.md# CLAUDE.md

## Visão Geral do Projeto
Sistema de e-commerce B2B para distribuidores industriais.
Arquitetura: monolito modular em Python com workers assíncronos.
Usuários são empresas (não pessoas físicas) com múltiplos compradores.

## Stack Técnica
- Linguagem: Python 3.12
- Framework: FastAPI 0.110 (async)
- ORM: SQLAlchemy 2.0 (async) + Alembic
- Banco de dados: PostgreSQL 16
- Cache/Fila: Redis 7 + Celery
- Testes: pytest + pytest-asyncio + httpx
- Linter/Formatter: ruff (substitui black, isort, flake8)
- Type checker: mypy --strict

## Comandos Importantes
- `make test`        — roda toda a suíte de testes
- `make test-unit`   — apenas testes unitários (rápido)
- `make lint`        — ruff check + mypy
- `make format`      — ruff format
- `make dev`         — inicia servidor em modo dev (uvicorn --reload)
- `make build`       — build da imagem Docker
- `docker compose up -d` — sobe Postgres + Redis locais
- `alembic upgrade head` — aplica migrações pendentes

## Convenções de Código
- snake_case para funções, variáveis e módulos
- PascalCase para classes e tipos
- UPPER_SNAKE_CASE para constantes
- Docstrings no formato Google
- Imports absolutos: `from app.services.order import ...`
- Type hints obrigatórios em funções públicas
- Linhas até 100 caracteres

## Arquitetura (camadas)
app/
├── routers/       ← Thin controllers (FastAPI routes)
├── schemas/       ← Pydantic v2 (request/response DTOs)
├── services/      ← Lógica de negócio
├── repositories/  ← Acesso a dados (Repository pattern)
├── models/        ← SQLAlchemy ORM models
├── workers/       ← Tarefas Celery
└── core/          ← Config, deps, security, logging

## Regras Críticas
- SEMPRE rode `make lint` antes de commitar
- NUNCA edite arquivos em `migrations/versions/` manualmente — use `alembic revision --autogenerate`
- NUNCA faça `commit` direto na branch `main`
- Testes devem usar fixtures do `conftest.py` para sessão de DB
- Segredos vão em `.env` (nunca commite)
- Queries com `select()` — evite SQL raw a menos que documentado

## Contexto de Negócio
- "Conta" = empresa compradora (pessoa jurídica)
- "Usuário" = comprador individual vinculado a uma Conta
- Pedidos podem ter múltiplos endereços de entrega (split-shipment)
- Preços são negociados por Conta (tabela `price_agreements`)
- Créditos B2B: pedidos podem ser aprovados sem pagamento imediato
💡
Problemas que resolve
  • Consistência entre todos os desenvolvedores do time (mesmas convenções aplicadas)
  • Contexto persistente entre sessões (não é preciso re-explicar o projeto)
  • Enforcement automático de convenções (naming, imports, padrões)
  • Onboarding instantâneo de novos desenvolvedores via IA
⚠
Boa prática

Mantenha o CLAUDE.md abaixo de 300 linhas. Arquivos muito longos consomem tokens de contexto e diluem sinal. Se precisar de mais, use CLAUDE.md em subdiretórios com escopo local.

📚
Referência

Anthropic. "Memory & CLAUDE.md". Disponível em: https://docs.anthropic.com/en/docs/claude-code/memory

8.4.2 GitHub Copilot — .github/copilot-instructions.md

O GitHub Copilot lê instruções customizadas a partir do repositório e as aplica tanto em sugestões inline quanto em respostas do Copilot Chat.

⚙ .github/copilot-instructions.md Arquivo principal (raiz do repositório)
⚙ .github/instructions/*.instructions.md Instruções segmentadas por caminho (path-specific)
ℹ
Limite prático

O Copilot suporta arquivos grandes (~128K caracteres), mas na prática instruções de 200–400 linhas funcionam melhor — contexto excessivo prejudica sugestões inline.

Template

.github/copilot-instructions.md# Copilot Instructions

## Project Context
Microserviço Go responsável pelo processamento de pagamentos (cartão de
crédito, PIX, boleto) para uma plataforma SaaS multi-tenant.

## Stack
- Go 1.22
- Echo v4 (HTTP framework)
- pgx v5 (driver Postgres, sem ORM)
- slog (logging estruturado)
- testify + httptest (testes)

## Coding Guidelines
- Prefira a biblioteca padrão quando possível
- Erros SEMPRE envoltos com contexto: `fmt.Errorf("order: %w", err)`
- Use `context.Context` como primeiro parâmetro em funções que fazem I/O
- Handlers HTTP ficam em `internal/http/handlers/`
- Lógica de negócio em `internal/domain/`
- NÃO use `interface{}` — prefira `any` ou tipos específicos

## Naming Conventions
- Pacotes: lowercase, sem underscore (`paymentgateway`, não `payment_gateway`)
- Arquivos: snake_case (`order_service.go`)
- Tipos exportados: PascalCase, não-exportados: camelCase
- Interfaces: sufixo `-er` quando possível (`Repository`, `Notifier`)

## Testing Expectations
- Table-driven tests para todo código de domínio
- Use `t.Run(tc.name, ...)` para subtests nomeados
- Mocks gerados com `mockery` em `internal/mocks/`
- Cobertura mínima: 80% em `internal/domain/`

## Security Considerations
- NUNCA logar dados sensíveis (PAN, CVV, senhas, tokens)
- Validar entrada em TODOS os handlers (usar `go-playground/validator`)
- Queries SQL sempre parametrizadas ($1, $2...) — nunca concatenação
- Segredos via variáveis de ambiente, nunca hardcoded
- Rate limiting obrigatório em rotas públicas
💡
Problemas que resolve
  • Sugestões inline seguem convenções do time (não apenas boas práticas genéricas)
  • Respostas do Copilot Chat são consistentes para todos os devs
  • Reforça regras de segurança que a IA poderia violar silenciosamente
📚
Referência

GitHub Docs. "Configure custom instructions for GitHub Copilot". Disponível em: https://docs.github.com/en/copilot/how-tos/configure-custom-instructions

8.4.3 Cursor — .cursor/rules/*.mdc (Project Rules)

O Cursor adotou um novo formato baseado em arquivos .mdc (Markdown com frontmatter YAML) que substitui o antigo arquivo único .cursorrules. Permite múltiplos conjuntos de regras com escopo granular.

📁 .cursor/rules/ Rules do projeto (commitadas)
⚙ typescript.mdc Regras para arquivos TypeScript
⚙ api-conventions.mdc Convenções de rotas e handlers
⚙ testing.mdc Padrões de teste (auto-attached)
📁 ~/.cursor/rules/ User Rules (pessoais, não commitadas)

Tipos de ativação (frontmatter)

Tipo Frontmatter Quando aplica
Always alwaysApply: true Em todo prompt, sempre
Auto Attached globs: [...] Quando arquivos que casam com o glob estão no contexto
Agent Requested description: "..." IA decide se aplica baseada na descrição
Manual (sem campos ativos) Só quando invocado com @ruleName

Template com frontmatter

.cursor/rules/typescript.mdc---
description: Regras de TypeScript strict e convenções de tipagem
globs:
  - "**/*.ts"
  - "**/*.tsx"
alwaysApply: false
---

# TypeScript Strict

## Tipagem
- `strict: true` no tsconfig — nunca desabilite
- NUNCA use `any`. Prefira `unknown` + narrowing ou tipos específicos
- Prefira `type` para uniões/intersecções; `interface` para contratos públicos
- Use `satisfies` em vez de type assertions (`as`) sempre que possível

## React
- Componentes sempre como funções (não classes)
- Props com `interface` nomeada: `interface UserCardProps { ... }`
- Hooks customizados começam com `use` (regra do React)
- Server Components por padrão no Next.js App Router

## Exemplos
// BAD
const data = response as User;

// GOOD
const data = response satisfies User;

// BAD
function fetch(opts: any) { ... }

// GOOD
function fetch(opts: FetchOptions) { ... }
💡
Problemas que resolve
  • Escopo granular: regras de front não aparecem quando você edita código de backend
  • Ativação por padrão de arquivo: regras de teste só são aplicadas quando você abre um *.test.ts
  • Múltiplos conjuntos: separe convenções por domínio (API, UI, testes, infra)
  • User Rules: preferências pessoais do dev sem poluir o projeto
📚
Referência

Cursor. "Rules". Disponível em: https://cursor.com/docs/rules

8.4.4 OpenAI Codex — AGENTS.md

O formato AGENTS.md foi proposto pela OpenAI (Codex) e vem sendo adotado como padrão de fato por múltiplas ferramentas de IA — incluindo Claude Code (como fallback), Cursor e outros. O arquivo mais próximo (nearest) vence.

⚙ ./AGENTS.md Raiz do projeto
⚙ ./packages/api/AGENTS.md Específico do subpacote (sobrescreve raiz)
⚙ ./packages/web/AGENTS.md Específico do frontend

Template

AGENTS.md# AGENTS.md

## Project Introduction
Plataforma SaaS multi-tenant para gestão de frotas. Monorepo com:
- `packages/api`  — backend Node.js + Fastify
- `packages/web`  — frontend Next.js 14
- `packages/shared` — tipos e utilitários compartilhados

## Tooling
- Gerenciador: pnpm workspaces
- Node: 20.x (use `.nvmrc`)
- Build: tsup (libs) + next build (app)
- Test: vitest (unit) + playwright (e2e)
- Lint: biome (substitui eslint + prettier)

## Build & Test Commands
- `pnpm install`           — instala dependências do monorepo
- `pnpm -r build`          — builda todos os pacotes
- `pnpm -r test`           — roda testes em todos os pacotes
- `pnpm --filter api dev`  — inicia apenas a API em dev
- `pnpm --filter web dev`  — inicia apenas o frontend em dev
- `pnpm lint`              — biome check em todo o repo
- `pnpm e2e`               — playwright (sobe docker-compose antes)

## Conventions
- Arquivos: kebab-case (`order-service.ts`)
- Exports nomeados (evite `export default` exceto em pages do Next)
- Schemas de validação com Zod, reutilizados entre server e client
- Commits seguem Conventional Commits (feat:, fix:, chore:, etc.)
- PRs precisam de 1 aprovação + CI verde

## Things to Avoid
- NÃO instale pacotes globais — sempre workspace-scoped
- NÃO importe de `packages/api` no frontend (use `packages/shared`)
- NÃO use `process.env` diretamente — use o módulo `config` validado com Zod
- NÃO faça consultas a DB no frontend (só via API)
- NÃO altere arquivos em `generated/` — são autogerados por Prisma
ℹ
Padrão cross-tool

AGENTS.md é reconhecido por Codex, Claude Code (fallback), Cursor, Aider e outros. Ter um único AGENTS.md evita manter N arquivos de rules sincronizados para cada ferramenta usada pelo time.

💡
Problemas que resolve
  • Compatibilidade entre ferramentas (um único arquivo, múltiplos agentes)
  • Fonte única de verdade para contexto do projeto
  • Hierarquia natural em monorepos (cada pacote com seu AGENTS.md)
📚
Referência

OpenAI Developers. "Codex & AGENTS.md". Disponível em: https://developers.openai.com/codex

8.4.5 Windsurf — .windsurf/rules/ ou .windsurfrules

O Windsurf (Codeium) suporta dois formatos: o legado .windsurfrules (arquivo único) e o moderno diretório .windsurf/rules/ com múltiplos arquivos e modos de ativação similares ao Cursor.

📁 .windsurf/rules/ Formato moderno (recomendado)
⚙ global.md Aplicado sempre
⚙ python.md Auto-attached para *.py
⚙ .windsurfrules Formato legado (arquivo único)

Template

.windsurf/rules/python.md---
trigger: glob
globs:
  - "**/*.py"
---

# Python Conventions

## Stack
- Python 3.12 + Poetry
- FastAPI 0.110 + SQLAlchemy 2.0 (async)
- pytest + pytest-asyncio
- ruff + mypy --strict

## Regras
- Type hints em TODAS as funções públicas
- Docstrings formato Google
- Prefira dataclasses/pydantic a dicts para estruturas
- NÃO use `print()` — use `logger.info()` (logging estruturado)
- Imports absolutos sempre (`from app.services import ...`)

## Padrões
- Repository pattern em `app/repositories/`
- Services em `app/services/` (lógica de negócio)
- Rotas finas em `app/routers/` (apenas wiring)

## Testes
- Fixtures em `tests/conftest.py`
- Nome: `test___`
- Use `pytest.mark.asyncio` para testes async
💡
Problemas que resolve
  • Modos de ativação (always/glob/manual) semelhantes ao Cursor
  • Separação clara entre regras globais e específicas de linguagem
  • Migração gradual do formato legado para o moderno

Seção 8.5

Problemas que Rules Resolvem

Entender por que rules existem é tão importante quanto saber como escrevê-las. Abaixo, os sete problemas concretos que sistemas de rules resolvem em projetos reais.

# Problema Como Rules resolvem
1 Inconsistência entre desenvolvedores Diferentes pessoas usam a IA, mas todas recebem as mesmas instruções automaticamente. O código gerado segue o mesmo estilo independente de quem prompta.
2 Repetição de contexto Informações do projeto (stack, arquitetura, convenções) não precisam ser re-explicadas a cada sessão. O custo cognitivo de "preparar o terreno" vai a zero.
3 Aderência a convenções Naming, padrões arquiteturais, estilo de código — a IA passa a seguir automaticamente, mesmo quando o desenvolvedor não lembra de mencionar.
4 Restrições explícitas "Não use a biblioteca X (deprecated)", "Não edite arquivos em migrations/", "Nunca logue PAN de cartão". Regras negativas que evitam erros sistemáticos.
5 Contexto de domínio Regras de negócio específicas que a IA não tem como inferir do código (ex: "Conta = empresa, não usuário; pedidos B2B podem ser split-shipment").
6 Onboarding Novos devs recebem a IA "pré-treinada" no projeto. Em vez de perguntar ao time, perguntam ao Claude/Copilot, que já conhece as convenções.
7 Evolução do projeto Rules refletem o estado atual do projeto. Quando convenções mudam, atualizar o arquivo atualiza o comportamento da IA para todo o time, imediatamente.
💡
Regra geral

Se você já teve que explicar a mesma coisa para a IA duas vezes, essa informação deveria estar em um arquivo de rules.

Seção 8.6

Boas Práticas Universais para Rules

Independentemente da ferramenta (Claude Code, Copilot, Cursor, Codex, Windsurf...), os princípios abaixo se aplicam a qualquer sistema de rules. Seguir estas práticas transforma rules de "documentação que a IA ignora" em "guard-rails que moldam consistentemente o output".

1. Seja específico, não genérico

Rules genéricas são ruído — a IA já sabe "boas práticas" em geral. Valor real vem de regras concretas e acionáveis, específicas do seu projeto.

✗ Genérico demais
Siga boas práticas de código.
Escreva código limpo.
Use padrões de projeto.
Trate erros adequadamente.
✓ Específico e acionável
Use Repository pattern: toda query vai em
app/repositories/, nunca em services/ ou routers/.

snake_case para funções, PascalCase para classes.

Erros de domínio: `raise DomainError("código: mensagem")`.
Erros HTTP: só em handlers, via HTTPException.

Sempre rode `make lint` antes de commitar.
Imports absolutos: `from app.services import ...`

2. Inclua exemplos de código

Uma convenção descrita em texto é ambígua; uma convenção com exemplo é inequívoca. Inclua snippets curtos mostrando a convenção em prática.

3. Mantenha atualizado

Rules desatualizadas são piores que não ter rules — induzem a IA a gerar código errado com confiança. Revisite o arquivo a cada mudança significativa de arquitetura ou convenção.

4. Commite no repositório

Rules devem ser versionadas junto com o código. Se o time todo vai usar, o arquivo vai no git. Exceções: arquivos locais não-commitados (ex: .claude/CLAUDE.md) para preferências pessoais.

5. Não duplique CLAUDE.md em README

README é para humanos (onboarding, como rodar, links). Rules são para a IA (convenções técnicas detalhadas, restrições). Duplicar ambos gera divergência. Audiências diferentes, conteúdos diferentes.

6. Priorize o que a IA erra

Se a IA sempre acerta algo sem precisar de rule, não escreva rule para isso — é token desperdiçado. Observe onde a IA erra repetidamente no seu projeto e crie rules para esses casos específicos.

7. Inclua comandos de build/test

A IA precisa saber como verificar seu trabalho. Liste comandos explícitos: make test, pnpm lint, cargo check. Sem isso, a IA adivinha ou roda comandos errados.

8. Use hierarquia quando apropriado

Em monorepos ou projetos grandes, aproveite a hierarquia: rules globais (stack, comandos gerais) na raiz; rules específicas (convenções de subpacote, regras de domínio) em subdiretórios. Evite um arquivo gigantesco.

💡
Dica prática

Comece com um arquivo de rules mínimo (stack, comandos, 3-5 convenções críticas). Adicione novas rules apenas quando identificar um padrão de erro da IA que se repete. Rules devem crescer organicamente, não serem escritas de uma vez.

Exercício: Audite suas próprias rules
  1. Abra seu arquivo de rules atual (CLAUDE.md, copilot-instructions.md, .cursor/rules/, AGENTS.md).
  2. Risque tudo que é genérico — frases como "siga boas práticas", "escreva código limpo", "use padrões apropriados". Se a IA já saberia sem você falar, corte.
  3. Liste 5 erros que a IA cometeu essa semana no seu projeto. Para cada erro, escreva uma rule específica que o preveniria.
  4. Verifique se tem comandos de build/test documentados. Se não, adicione.
  5. Confira se há exemplos de código. Convenções sem exemplo são ambíguas — converta pelo menos 2 em snippets curtos.
  6. Meça a quantidade de linhas. Acima de 300? Considere dividir em arquivos específicos por subdiretório ou tipo de arquivo.
  7. Commite as mudanças e peça ao time para fazer o mesmo exercício no próximo sprint.
