Agentes Autônomos e Sistemas Multi-Agentes

O loop ReAct, padrões arquiteturais multi-agentes, benchmarks SWE-bench e o futuro do desenvolvimento com IA.

Seção 13.1

O que é um Agente de IA?

Um agente de IA é um sistema que usa um LLM como "cérebro" para planejar, executar ações e observar resultados de forma autônoma, iterando até completar uma tarefa.

Diferença fundamental

Chatbot (sem agência)
Usuário pergunta → Modelo responde → Fim
Interação única, sem autonomia
Agente (com agência)
Usuário dá tarefa → Modelo PLANEJA
Modelo EXECUTA → Modelo OBSERVA resultado
Modelo decide → PRÓXIMA AÇÃO
... loop → Resultado final
Loop autônomo até completar a tarefa

O loop do agente (ReAct pattern)

RACIOCÍNIO (Reasoning)
"Preciso primeiro ler o arquivo para entender o bug"
AÇÃO (Action)
Read("src/auth.py")
OBSERVAÇÃO (Observation)
"O arquivo contém uma condição incorreta na linha 45"
RACIOCÍNIO (Reasoning)
"Agora vou corrigir a condição e rodar os testes"
... (loop continua)
loop continua até a tarefa ser concluída
📚

  • Anthropic. "Building effective agents". anthropic.com/research/building-effective-agents — Guia prático da Anthropic sobre como construir agentes eficazes.

Seção 13.3

Padrões Arquiteturais de Agentes

11.3.1 Agente Único (Single Agent)

O padrão mais simples — um único agente com acesso a ferramentas:

Usuário
→
Agente + Ferramentas
→
Resultado

Exemplo: Claude Code no modo padrão. Um único agente lê, edita, busca e executa.

Quando usar: Tarefas focadas que um único "desenvolvedor" poderia resolver.

11.3.2 Multi-Agentes em Paralelo

Múltiplos agentes trabalham simultaneamente em subtarefas independentes:

Usuário
→
Orquestrador
→
Agente 1: Frontend
Agente 2: Backend
Agente 3: Testes
→
Resultado combinado

Exemplo: Claude Code com sub-agentes (Agent tool). O agente principal delega:

  • Um sub-agente pesquisa a codebase
  • Outro sub-agente roda em um worktree separado
  • Resultados são combinados pelo agente principal

Quando usar: Tarefas que podem ser decompostas em partes independentes.

11.3.3 Pipeline de Agentes

Agentes encadeados onde a saída de um alimenta o próximo:

Análise
→
Implementação
→
Review
→
Teste

Quando usar: Workflows com etapas naturalmente sequenciais.

11.3.4 Agente Supervisor

Um agente coordenador que distribui e supervisiona trabalho:

Supervisor
→
Worker 1
Worker 2
Worker 3
→
Supervisor valida

Quando usar: Tarefas complexas que precisam de coordenação e validação.

Referência

Wang, L. et al. (2024). "A Survey on Large Language Model based Autonomous Agents". Renmin University. arXiv:2308.11432

Seção 13.4

Subagentes: Delegação Inteligente de Tarefas

Subagentes são agentes de IA especializados que são invocados por um agente principal para executar subtarefas específicas. Cada subagente opera com sua própria janela de contexto isolada, recebe um prompt customizado, tem acesso a ferramentas específicas e retorna apenas um resumo ao agente pai.

Analogia

Imagine um gerente de projeto (agente principal) que, em vez de fazer tudo sozinho, delega tarefas para membros especializados da equipe (subagentes). O gerente diz "pesquise como funciona a autenticação" para o analista, "refatore o módulo de pagamentos" para o desenvolvedor sênior, e "escreva testes para a API" para o QA. Cada membro trabalha independentemente, em sua própria mesa (contexto), e reporta apenas o resultado final.

Por que Subagentes existem?

O problema fundamental que subagentes resolvem é a limitação da janela de contexto (Módulo 3). Sem subagentes:

Sem Subagentes
Toda pesquisa polui o contexto principal
Resultados intermediários consomem tokens
Tarefas sequenciais (uma por vez)
Contexto esgota rapidamente
Capacidade limitada, custo alto
Com Subagentes
Cada subagente tem contexto próprio
Apenas resumos voltam ao pai
Tarefas em paralelo (simultâneas)
Contexto principal permanece limpo
Capacidade multiplicada, custo otimizado

Como Subagentes Funcionam

O ciclo de vida de um subagente segue este fluxo:

1. Agente principal identifica subtarefa
"Preciso pesquisar como a autenticação funciona antes de corrigir o bug"
2. Cria subagente com prompt específico
Define: tipo, prompt, ferramentas permitidas, modelo, modo de isolamento
3. Subagente trabalha em contexto isolado
Usa sua própria janela de contexto — lê arquivos, executa comandos, analisa código
4. Retorna apenas o resumo ao agente pai
Resultados intermediários (tool calls, arquivos lidos) NÃO voltam — apenas a conclusão
5. Agente principal continua com contexto limpo
Incorpora o resumo e prossegue com o contexto principal intacto

Isolamento de Janela de Contexto

A propriedade mais importante de um sistema de subagentes bem projetado é o isolamento de escopo. Cada subagente opera em sua própria janela de contexto, completamente separada do agente pai e de outros subagentes paralelos.

🔎
Conceito-Chave: Isolamento de Contexto

Quando um subagente é criado, ele recebe apenas o prompt do agente pai — nada mais. Ele não herda o histórico da conversa, nem os arquivos já lidos. Isso significa que o subagente começa "do zero" com um contexto limpo, e o agente pai não é poluído com os resultados intermediários do subagente.

Quais ferramentas suportam subagentes com contexto isolado?

Ferramenta Isolamento de Contexto Paralelo Aninhamento Detalhes
Claude Code Completo Sim Não (1 nível) Cada subagente tem janela própria. Pai recebe apenas resumo.
Cursor Completo Sim (até 8 background) Sim (desde v2.5) Subagentes e Background Agents com contexto isolado.
OpenAI Codex Completo Sim (até 6 threads) Profundidade 1 (padrão) Contexto independente por thread. Inter-agent messaging v2.
Devin Completo Sim (múltiplos Devins) N/A Cada Devin roda em VM cloud separada com IDE própria.

Criando Subagentes Customizados

Subagentes customizados são definidos como arquivos Markdown com frontmatter YAML em .claude/agents/:

.claude/agents/security-reviewer.md---
name: Security Reviewer
description: Revisa código focando em vulnerabilidades OWASP Top 10
model: sonnet
tools:
  - Read
  - Grep
  - Glob
maxTurns: 15
---

# Instruções

Você é um revisor de segurança especializado. Ao receber código:

1. Busque por vulnerabilidades OWASP Top 10
2. Verifique: SQL injection, XSS, CSRF, auth bypass
3. Analise dependências por CVEs conhecidas
4. Produza relatório com severidade e correções

Nunca modifique arquivos — apenas analise e reporte.

Exemplo Prático: Múltiplos Subagentes em Paralelo

Claude Code# O agente principal pode lançar vários subagentes simultaneamente:

Tarefa: "Implemente o sistema de notificações"

Agente Principal decide paralelizar:
├─ Agent(type="Explore")    # Pesquisa: como o sistema de eventos funciona
├─ Agent(type="Explore")    # Pesquisa: quais notificações já existem
└─ Agent(type="Plan")       # Planeja: arquitetura da feature

# Cada um trabalha em paralelo com contexto próprio
# Pai recebe 3 resumos concisos
# Então implementa com base nos resumos combinados

Seção 13.5

Worktrees: Isolamento Seguro para Trabalho Paralelo

Uma worktree (árvore de trabalho) é um recurso do Git que permite ter múltiplas cópias de trabalho do mesmo repositório simultaneamente, cada uma em um branch diferente, sem precisar clonar o repositório inteiro. No contexto de agentes de IA, worktrees são a tecnologia que permite que subagentes façam mudanças no código sem afetar o diretório de trabalho principal.

Analogia

Imagine que seu repositório é um documento original. Em vez de editá-lo diretamente (arriscando estragar algo), você faz uma cópia de trabalho (worktree). Se as mudanças ficarem boas, você incorpora a cópia. Se ficarem ruins, simplesmente joga a cópia fora — o original permanece intacto.

O Problema que Worktrees Resolvem

Sem worktrees, quando um agente de IA edita arquivos no seu projeto, as mudanças acontecem diretamente no seu diretório de trabalho. Isso causa problemas:

  • Risco de corrupção: se o agente fizer mudanças ruins, seu código fica em estado inconsistente
  • Bloqueio: enquanto o agente trabalha, você não pode continuar editando
  • Sem paralelismo: dois agentes não podem modificar os mesmos arquivos simultaneamente
  • Rollback difícil: desfazer mudanças parciais é complicado

Como Worktrees Funcionam com Agentes

1. Agente cria worktree
Git cria um diretório separado com cópia do repo em um branch temporário
git worktree add /tmp/worktree-abc feature-branch
2. Subagente trabalha no worktree
Todas as edições, criações e exclusões de arquivos acontecem no diretório isolado
3a. Sucesso: mudanças preservadas
O worktree é mantido com o path e branch retornados ao agente pai para review/merge
3b. Falha: worktree descartado
Se o subagente não fez mudanças (ou falhou), o worktree é automaticamente limpo
4. Diretório principal intacto
Em nenhum momento o código no seu diretório de trabalho foi modificado

Worktrees no Claude Code

No Claude Code, worktrees são ativadas através do parâmetro isolation: "worktree" na ferramenta Agent:

Claude Code# Subagente com worktree para refatoração segura:
Agent(
  description: "Refatorar módulo de pagamentos",
  prompt: "Refatore payments/ de sync para async/await...",
  isolation: "worktree"
)

# Resultado (se houve mudanças):
# {
#   worktree_path: "/tmp/worktree-a1b2c3",
#   branch: "agent/refactor-payments-a1b2c3",
#   summary: "Convertidos 12 arquivos para async..."
# }

Cenários Ideais para Worktrees

Cenário Por que usar worktree Exemplo
Refatoração grande Mudanças em muitos arquivos — seguro testar antes de mergear Migrar de ORM, trocar framework de testes
Experimentação Testar uma abordagem sem comprometer o código atual "Tente implementar isso com Redis, e se não funcionar, descarte"
Trabalho paralelo Múltiplos agentes trabalhando simultaneamente no mesmo repo Um agente no frontend, outro no backend, outro nos testes
Code review seguro Agente aplica sugestões de review sem tocar no branch principal "Aplique todas as sugestões do PR review em um worktree"
Prototipação Criar PoC rápida que pode ser descartada "Crie um protótipo de como ficaria com GraphQL"

Worktrees + Subagentes: O Poder Combinado

A combinação de subagentes com worktrees permite o padrão mais poderoso de desenvolvimento com IA — múltiplos agentes trabalhando em paralelo, cada um em seu próprio branch isolado:

Tarefa: "Implementar feature X"
→
Agente Principal
→
Subagente 1 (worktree A) — Backend API
Subagente 2 (worktree B) — Frontend UI
Subagente 3 (worktree C) — Testes E2E
→
Merge dos 3 branches
💡
Benefícios Combinados
  • Contexto isolado: cada subagente tem sua própria janela de contexto (não polui o pai)
  • Filesystem isolado: cada worktree tem sua própria cópia dos arquivos (não conflita)
  • Paralelismo real: 3 agentes trabalhando simultaneamente = 3x throughput
  • Rollback trivial: descarte qualquer worktree sem afetar os outros
  • Review facilitado: cada branch pode ser revisado independentemente antes do merge

Ferramentas que Suportam Worktrees

Ferramenta Suporte a Worktree Como funciona
Claude Code Nativo isolation: "worktree" na ferramenta Agent. Cleanup automático se sem mudanças.
Cursor Via Background Agents Background Agents clonam o repo na cloud e trabalham em branch separado.
OpenAI Codex Sandbox cloud Cada agente roda em sandbox cloud preloaded com o repositório.
Devin VM isolada Cada Devin opera em VM cloud separada com IDE completa.

Seção 13.7

O Futuro dos Agentes de Codificação

Tendências observadas e projetadas:

  1. Agentes cada vez mais autônomos — De "sugerir código" para "implementar features completas"
  2. Integração com CI/CD — Agentes que automaticamente criam PRs, rodam testes e fazem deploy
  3. Multi-agentes especializados — Times virtuais com agentes para frontend, backend, QA, DevOps
  4. Agentes com memória de longo prazo — Aprendendo preferências e padrões do projeto ao longo do tempo
  5. Agentes proativos — Agentes que monitoram repositórios e propõem melhorias automaticamente
  6. Verificação formal — Uso de provas matemáticas para garantir correção do código gerado
💡
Visão de Futuro

Estamos vivendo uma transição de "IA como autocomplete" para "IA como colega de equipe". Em breve, a pergunta não será "a IA sabe programar?", mas sim "como eu orquestro um time de agentes para entregar o melhor software possível?"
