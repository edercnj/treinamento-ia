LLMOps e Observabilidade em Produção

A maioria dos projetos de IA falha na operação, não na escolha do modelo. Tracing, prompt versioning, guardrails em runtime, rate limits, fallbacks e migração de modelos.

Seção 17.1

LLMOps vs. MLOps

MLOps clássico lida com modelos próprios, treinados e versionados como artefatos. LLMOps adiciona complicações específicas:

  • O "modelo" é uma API externa que o provedor muda sem avisar.
  • O artefato mais importante é o prompt, não o modelo.
  • Custo varia por token, não por compute alocado.
  • Latência é imprevisível (reasoning pode levar minutos).
  • Deprecação de modelos é frequente (3–12 meses).

Seção 17.3

Prompt Registry e versionamento

  • Prompt registry externo (LangSmith Prompts, Langfuse Prompts) — rollback sem deploy.
  • MDX/YAML no repo — versionamento via git, diff legível.
  • BAML / DSPy — declarativo, compila prompts de schemas e exemplos.
📚
Referências

Khattab, O. et al. (2023). "DSPy: Compiling Declarative Language Model Calls into Self-Improving Pipelines". Stanford, arXiv:2310.03714.

Seção 17.4

Guardrails em runtime

  • Llama Guard — Meta, classificador open source. Inan 2023, arXiv:2312.06674.
  • NVIDIA NeMo Guardrails — Colang DSL. Rebedea 2023, arXiv:2310.10501.
  • Guardrails AI — validadores composáveis em Python.
  • Anthropic Constitutional Classifiers — filtros especializados.
  • OpenAI Moderation API — classificador gratuito.
  • Promptfoo red-teaming — adversarial testing automatizado.
Seção 17.5

Rate limits, fallbacks e resiliência

  1. Exponential backoff com jitter — nunca retry imediato.
  2. Circuit breakers — após N falhas, pare por T segundos.
  3. Cascata de provedores — Anthropic → OpenAI → self-hosted.
  4. Roteamento inteligente — LiteLLM, OpenRouter.
  5. Provisioned throughput — Anthropic, Google, AWS Bedrock.
Seção 17.6

SLOs, latência e p99

Métricas essenciais
  • TTFT (Time To First Token) — quando o stream começa.
  • TPS (Tokens Per Second) — velocidade após TTFT.
  • E2E latency — prompt in a resposta out.
  • Error rate por categoria (timeout, 429, 5xx, guardrail block).

Modelos de reasoning têm cauda pesada de latência (10–60s). Streaming mitiga latência percebida.

Seção 17.7

Migração entre modelos

  1. Eval suite completo no modelo antigo (baseline).
  2. Mesmo eval no modelo novo.
  3. Identificar regressões por categoria.
  4. A/B test com tráfego real (5% → 50% → 100%).
  5. Manter modelo antigo como fallback por 1 ciclo.
  6. Atualize prompts — prompts de um modelo raramente são ótimos em outro.
Seção 17.8

Custo em produção

  • Atribuição por request/usuário/feature/tenant.
  • Alertas por desvio (ex: request usou 10× o p95 de tokens).
  • Budget enforcement (hard caps por org).
  • Reporting semanal/mensal com breakdown por modelo.
