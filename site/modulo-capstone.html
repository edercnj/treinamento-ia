Capstone: Agente de Revisão de PRs

Projeto integrativo que amarra todos os módulos anteriores. Arquitetura, iterações, golden set, riscos e evolução de um agente de revisão de Pull Requests.

Objetivo

Amarrar tudo em um sistema real

Este módulo descreve a arquitetura e iterações de um agente que recebe um Pull Request, analisa o diff, executa testes, sugere melhorias e posta review no GitHub. Integra os 21 módulos anteriores.

22.1

Requisitos e escopo

Funcional
  • Receber webhook de PR aberto no GitHub.
  • Buscar diff, contexto de arquivos modificados e descrição.
  • Análise multi-passo: segurança, performance, legibilidade, testes.
  • Postar comentários in-line e sumário geral.
  • Sugerir edições concretas quando aplicável.
Não-funcional
  • Custo ≤ $0.30 por PR.
  • Latência ≤ 2 min para PRs < 300 LOC.
  • Zero credenciais expostas; GitHub App.
  • Eval contínua: 50 PRs históricos como golden set.
22.2

22.3

Stack sugerido

  • Orquestração: LangGraph ou Python direto (transparência).
  • LLMs: Haiku 4.5 (triagem), Sonnet 4.6 (review), Opus 4.6 opcional para críticos.
  • Observability: Langfuse ou LangSmith.
  • MCP servers: GitHub MCP, filesystem MCP (sandbox).
  • Guardrails: Llama Guard + validação custom.
  • Evals: Inspect ou promptfoo com 50 PRs rotulados.
22.4

22.5

Golden set de avaliação

50 PRs históricos com labels:

  • True positive: problemas reais corrigidos pós-review.
  • True negative: PRs limpos aprovados sem alteração.
  • False positive (alvo: < 15%): sugestões ignoradas pelo autor.
  • Latência e custo por PR.

Rode o eval a cada mudança de prompt/modelo/tool.

22.6

Riscos e mitigações

RiscoMitigação
Prompt injection em PR maliciosoSandbox, guardrails, isolamento
Vazamento de códigoZero-retention tier OU Bedrock OU self-hosted Llama 4
Custo descontroladoCap por repo/dia, alertas, circuit breaker
Sugestões ruins geram ruídoThreshold de confiança, revisor humano
Modelo deprecadoLiteLLM/OpenRouter, fallback configurado
22.7

Entrega e iteração

  1. Dogfooding — use em seu próprio repo por 2 semanas.
  2. Feedback — survey NPS após cada review.
  3. Medição — produtividade antes/depois (ref: Peng et al. 2023 sobre Copilot, arXiv:2302.06590).
22.8

Evolução sugerida

Depois que o capstone básico funcionar:

  • Multi-agente: um agente para segurança, outro para performance, orquestrador.
  • Aprendizado contínuo: reviews aceitas alimentam fine-tuning do Haiku.
  • Expansão: revisão de RFCs, specs, docs.
