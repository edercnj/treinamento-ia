Seção 11.1

Hooks: Automação Baseada em Eventos

Hooks são comandos shell que executam automaticamente em resposta a eventos específicos do Claude Code. Eles permitem criar automações que rodam antes ou depois de ações da IA.

Analogia

Hooks são como git hooks (pre-commit, post-commit) ou event listeners em JavaScript — código que executa automaticamente quando algo acontece.

Eventos disponíveis para Hooks no Claude Code

Evento Quando dispara Uso típico
PreToolUse Antes de qualquer ferramenta ser executada Validação, aprovação automática, bloqueio
PostToolUse Depois de uma ferramenta ser executada Logging, formatação automática, notificação
Notification Quando Claude precisa notificar o usuário Integração com Slack, email, etc.
Stop Quando Claude termina uma resposta Verificação final, cleanup

Configuração de Hooks (em settings.json)

JSON{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit",
        "command": "echo 'Arquivo sendo editado: $CLAUDE_FILE_PATH'"
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Write",
        "command": "npx prettier --write $CLAUDE_FILE_PATH 2>/dev/null || true"
      },
      {
        "matcher": "Bash",
        "command": "echo '[LOG] Comando executado: $CLAUDE_COMMAND'"
      }
    ],
    "Stop": [
      {
        "command": "make lint 2>&1 | tail -5"
      }
    ]
  }
}

Exemplos práticos de Hooks

1. Auto-formatação após edição:

JSON{
  "PostToolUse": [
    {
      "matcher": "Edit|Write",
      "command": "npx prettier --write \"$CLAUDE_FILE_PATH\" 2>/dev/null; ruff format \"$CLAUDE_FILE_PATH\" 2>/dev/null; true"
    }
  ]
}

2. Bloquear edição em arquivos protegidos:

JSON{
  "PreToolUse": [
    {
      "matcher": "Edit",
      "command": "if echo \"$CLAUDE_FILE_PATH\" | grep -q 'migrations/'; then echo 'BLOCKED: Não edite migrations diretamente'; exit 1; fi"
    }
  ]
}
⚠
Atenção

O hook acima usa exit 1 para bloquear a ação. Quando um hook PreToolUse retorna código de erro, a ferramenta não é executada.

3. Notificação no terminal quando uma tarefa longa termina:

JSON{
  "Stop": [
    {
      "command": "osascript -e 'display notification \"Claude Code terminou a tarefa\" with title \"Claude Code\"'"
    }
  ]
}

11.1.1 Eventos de Ciclo de Vida (Expandido)

Além dos eventos básicos, o Claude Code expõe outros eventos úteis para automação avançada:

Evento Quando dispara Uso típico
PreToolUse Antes de qualquer ferramenta ser executada Validação, aprovação automática, bloqueio de comandos perigosos
PostToolUse Depois de uma ferramenta ser executada Formatação automática, logging, typecheck incremental
Notification Quando o Claude precisa notificar o usuário Slack, e-mail, desktop notifications
Stop Quando o Claude termina uma resposta Cleanup, geração de relatório, notificação de término
SessionStart No início de uma sessão Context injection (branch atual, status do CI, tickets ativos)
SubagentStop Quando um subagente termina Consolidação de resultados, logs específicos de subagentes

Variáveis de ambiente disponíveis nos hooks

Variável Descrição
$CLAUDE_FILE_PATHCaminho do arquivo sendo editado (Edit/Write)
$CLAUDE_COMMANDComando sendo executado (Bash)
$CLAUDE_TOOL_NAMENome da ferramenta invocada (Edit, Write, Bash, ...)
$CLAUDE_SESSION_IDID da sessão atual
$CLAUDE_PROJECT_DIRDiretório raiz do projeto

Onde configurar hooks

Arquivo Escopo Versionado em git?
~/.claude/settings.jsonGlobal (todas as sessões do usuário)Não (é pessoal)
.claude/settings.jsonProjeto (compartilhado com o time)Sim
.claude/settings.local.jsonProjeto, só localNão (adicione ao .gitignore)

11.1.2 Hooks em Outras Ferramentas

  • Cursor: possui background tasks e hooks baseados em MCP, mas com flexibilidade menor do que o Claude Code (menos eventos, menos variáveis de contexto).
  • GitHub Copilot: não expõe hooks configuráveis pelo usuário. A customização é feita apenas via instruções textuais.
  • OpenAI Codex: eventos de sandbox limitados; suporte a hooks é restrito a pre/post execução em contextos controlados.
  • Windsurf: ações encadeadas em workflows, mas sem o modelo de event-driven hooks.
💡
Nota

Hooks de ciclo de vida profundos e configuráveis são, hoje, amplamente um diferencial do Claude Code. Se automação baseada em eventos é crítica para o seu time, essa é uma razão forte de adoção.

11.1.3 Problemas que Hooks Resolvem

A seguir, oito problemas reais com exemplo concreto para cada um:

1. Formatação automática após edição

.claude/settings.json{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "command": "if [[ \"$CLAUDE_FILE_PATH\" =~ \\.py$ ]]; then ruff format \"$CLAUDE_FILE_PATH\" && ruff check --fix \"$CLAUDE_FILE_PATH\"; fi"
      }
    ]
  }
}

2. Guardrails de segurança — bloquear comandos perigosos

.claude/settings.json{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "command": "if echo \"$CLAUDE_COMMAND\" | grep -Eq '(rm -rf /|sudo rm|mkfs|:\\(\\)\\{)'; then echo 'BLOQUEADO: comando potencialmente destrutivo' >&2; exit 1; fi"
      }
    ]
  }
}

3. Auditoria e logging

.claude/settings.json{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": ".*",
        "command": "mkdir -p ~/.claude/logs && echo \"[$(date -Iseconds)] session=$CLAUDE_SESSION_ID tool=$CLAUDE_TOOL_NAME file=$CLAUDE_FILE_PATH cmd=$CLAUDE_COMMAND\" >> ~/.claude/logs/audit.log"
      }
    ]
  }
}

4. Validação contínua — typecheck após mudança em TS

.claude/settings.json{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "command": "if [[ \"$CLAUDE_FILE_PATH\" =~ \\.(ts|tsx)$ ]]; then npx tsc --noEmit 2>&1 | tail -20 || true; fi"
      }
    ]
  }
}

5. Notificação em Slack quando uma tarefa longa termina

.claude/settings.json{
  "hooks": {
    "Stop": [
      {
        "command": "curl -s -X POST -H 'Content-type: application/json' --data '{\"text\":\"Claude Code terminou a tarefa em '\"$CLAUDE_PROJECT_DIR\"'\"}' \"$SLACK_WEBHOOK_URL\" > /dev/null"
      }
    ]
  }
}

6. Integração com CI — disparar pipeline após mudança

.claude/settings.json{
  "hooks": {
    "Stop": [
      {
        "command": "cd \"$CLAUDE_PROJECT_DIR\" && if ! git diff --quiet; then gh workflow run ci.yml --ref \"$(git branch --show-current)\" || true; fi"
      }
    ]
  }
}

7. Context injection no início da sessão

.claude/settings.json{
  "hooks": {
    "SessionStart": [
      {
        "command": "echo \"Branch: $(git branch --show-current) | Último commit: $(git log -1 --oneline) | Status CI: $(gh run list -L 1 --json status -q '.[0].status' 2>/dev/null || echo 'n/a')\""
      }
    ]
  }
}

8. Compliance — forçar revisão para arquivos sensíveis

.claude/settings.json{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "command": "if echo \"$CLAUDE_FILE_PATH\" | grep -Eq '(auth|payments|crypto|secrets)/'; then echo 'ATENÇÃO: arquivo sensível — requer revisão humana antes do commit' >&2; fi"
      }
    ]
  }
}

11.1.4 Templates Prontos de Hooks (copy/paste)

Templates completos e prontos para uso. Adicione ao .claude/settings.json do seu projeto:

Template 1 — Python auto-format (ruff)

settings.json{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "command": "[[ \"$CLAUDE_FILE_PATH\" =~ \\.py$ ]] && ruff format \"$CLAUDE_FILE_PATH\" && ruff check --fix \"$CLAUDE_FILE_PATH\" || true"
      }
    ]
  }
}

Template 2 — TypeScript auto-format (prettier + eslint)

settings.json{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "command": "if [[ \"$CLAUDE_FILE_PATH\" =~ \\.(ts|tsx|js|jsx)$ ]]; then npx prettier --write \"$CLAUDE_FILE_PATH\" && npx eslint --fix \"$CLAUDE_FILE_PATH\" || true; fi"
      }
    ]
  }
}

Template 3 — Bloquear comandos perigosos

settings.json{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "command": "if echo \"$CLAUDE_COMMAND\" | grep -Eq '^(sudo rm|rm -rf /|rm -rf \\*|chmod -R 777|mkfs|dd if=)'; then echo 'BLOQUEADO: comando destrutivo detectado' >&2; exit 1; fi"
      }
    ]
  }
}

Template 4 — Bloquear edições em migrations

settings.json{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit",
        "command": "if echo \"$CLAUDE_FILE_PATH\" | grep -Eq '(migrations|alembic/versions|prisma/migrations)/'; then echo 'BLOQUEADO: migrations não devem ser editadas — crie uma nova migration' >&2; exit 1; fi"
      }
    ]
  }
}

Template 5 — Notificação Slack ao terminar (via webhook)

settings.json{
  "hooks": {
    "Stop": [
      {
        "command": "[ -n \"$SLACK_WEBHOOK_URL\" ] && curl -s -X POST -H 'Content-type: application/json' --data \"{\\\"text\\\":\\\"Claude Code finalizou em $(basename \"$CLAUDE_PROJECT_DIR\")\\\"}\" \"$SLACK_WEBHOOK_URL\" > /dev/null || osascript -e 'display notification \"Claude Code terminou\" with title \"Claude Code\"'"
      }
    ]
  }
}

Template 6 — Log de todas as tool uses com timestamp

settings.json{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": ".*",
        "command": "mkdir -p ~/.claude/logs && printf '[%s] session=%s tool=%s file=%s cmd=%s\\n' \"$(date -Iseconds)\" \"$CLAUDE_SESSION_ID\" \"$CLAUDE_TOOL_NAME\" \"${CLAUDE_FILE_PATH:-}\" \"${CLAUDE_COMMAND:-}\" >> ~/.claude/logs/tools-$(date +%Y-%m-%d).log"
      }
    ]
  }
}

11.1.5 Quando NÃO Usar Hooks

⚠
Anti-padrões com Hooks
  • Lógica de negócio: hooks são para automação de infraestrutura/qualidade. Lógica de produto pertence a skills ou ao próprio prompt.
  • Hooks que bloqueiam com frequência: se um hook PreToolUse dispara exit 1 em cada ação, o fluxo fica insuportável. Prefira alertas (stderr sem exit).
  • Side-effects irreversíveis: nunca coloque commits automáticos, deploys ou git push em hooks. Humanos devem autorizar isso.
  • Dependência síncrona de rede: um webhook lento pode travar toda a sessão. Use curl ... & (background) ou timeout curto.
  • Hooks longos: se o comando demora > 5s, o usuário sente. Faça o hook delegar para background.

11.1.6 Segurança em Hooks

🔒
Cuidado — hooks rodam com suas permissões
  • Hooks são executados com as mesmas permissões do usuário. Um hook malicioso pode acessar qualquer arquivo que você acessa.
  • Nunca interpole dados não-sanitizados em shell. "$CLAUDE_COMMAND" pode conter caracteres especiais — sempre entre aspas duplas.
  • Valide scripts com shellcheck antes de commitar hooks.
  • Em scripts bash usados por hooks, prefira set -euo pipefail para falhar cedo.
  • Revise hooks de terceiros (ex.: copiados da internet) antes de aplicar — eles rodam automaticamente.
  • Não commit .claude/settings.local.json — inclua-o no .gitignore.
hook-seguro.sh#!/usr/bin/env bash
set -euo pipefail

# Sempre cite variáveis
file="${CLAUDE_FILE_PATH:-}"
[ -z "$file" ] && exit 0

# Valide antes de usar
case "$file" in
  *.py) ruff format "$file" ;;
  *.ts|*.tsx) npx prettier --write "$file" ;;
  *) : ;;  # noop para outros tipos
esac
Seção 11.2

Skills + Hooks: Workflows Poderosos

A combinação de Skills e Hooks permite criar workflows sofisticados:

1. Usuário invoca: /deploy staging
2. Skill "deploy" executa
a. Roda testes (make test)
b. Verifica lint (make lint)
c. Cria tag de versão
d. Gera changelog
3. Hook PreToolUse (Bash)
🚫 Bloqueia qualquer comando rm -rf
🚫 Bloqueia deploy em produção sem flag explícita
4. Hook PostToolUse (Bash)
✔ Envia notificação no Slack
✔ Registra log de deploy
5. Hook Stop
✅ Verifica se testes passaram
📋 Gera relatório de deploy
📊
Ciclo de Vida

O fluxo completo de uma ação no Claude Code segue o ciclo: Evento → Hook PreToolUse → Execução da Tool → Hook PostToolUse → Resultado. Hooks permitem interceptar qualquer ponto deste ciclo.

📚
Referência

Anthropic. "Claude Code — Hooks". Disponível em: https://docs.anthropic.com/en/docs/claude-code/hooks

Seção 11.3

Comparativo Final: Skills vs Hooks vs Rules

Três conceitos estendem o comportamento de um agente de IA, mas cada um tem um papel distinto. Entender a diferença evita misturar responsabilidades.

Conceito Quando dispara Controlado por Exemplo
Rules Automaticamente, carregadas como contexto no início da sessão Arquivo Markdown no repositório (CLAUDE.md, .cursor/rules/) "Use sempre Portuguese em mensagens de commit"
Skills Quando o usuário invoca explicitamente (ou o modelo decide usar) Comando manual /skill ou seleção do modelo /commit, /review, /deploy
Hooks Eventos do sistema (PreToolUse, PostToolUse, Stop, ...) Configuração em settings.json Auto-format após edição, bloqueio de rm -rf

Ordem de execução em uma sessão típica

1. Rules carregadas no início da sessão
O conteúdo de CLAUDE.md e demais arquivos de regras entra no contexto inicial.
Hook SessionStart dispara
(opcional) Injeta contexto dinâmico — branch, status do CI, tickets ativos.
2. Usuário invoca uma Skill (ou conversa normal)
Ex.: /commit carrega o prompt especializado.
3. Claude decide executar uma tool
Ex.: Bash("git diff").
4. Hook PreToolUse dispara
Pode bloquear a execução retornando exit 1.
5. A tool executa
6. Hook PostToolUse dispara
Formatação, log, typecheck — tudo aqui.
7. Claude continua (loop) ou termina
8. Hook Stop dispara no fim
Notificação Slack, cleanup, relatório final.
📊
Regra prática

Se é algo que sempre deve ser verdade → Rule. Se é algo que o usuário invoca → Skill. Se é algo que deve rodar automaticamente em resposta a um evento → Hook.
