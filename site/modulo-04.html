Engenharia de Contexto

Os 4 pilares para preencher a janela de contexto da forma mais eficaz possível: seleção, organização, compressão e atualização dinâmica.

Seção 4.1

O que é Engenharia de Contexto?

Se a Janela de Contexto é o espaço disponível, a Engenharia de Contexto (Context Engineering) é a arte e ciência de preencher esse espaço da forma mais eficaz possível.

Definição formal: Engenharia de Contexto é o conjunto de técnicas, estratégias e decisões arquiteturais para selecionar, organizar, comprimir e gerenciar as informações que são enviadas a um modelo de linguagem, maximizando a qualidade das respostas dentro dos limites da janela de contexto.

Enquanto Prompt Engineering foca em como formular a pergunta, Engenharia de Contexto foca em o que incluir junto com a pergunta.

Analogia

Se você está consultando um advogado (a LLM):

  • Prompt Engineering = como formular sua pergunta ao advogado
  • Engenharia de Contexto = quais documentos, contratos e evidências levar à reunião

Levar documentos errados ou irrelevantes é tão prejudicial quanto não levar documentos suficientes.

Seção 4.2

Os Quatro Pilares da Engenharia de Contexto

Pilar 1 — Seleção de Contexto

Decidir o que incluir na janela de contexto é a decisão mais impactante:

Técnicas de seleção

Técnica Descrição Quando usar
Seleção manual O programador decide quais arquivos incluir Debugging focado, quando você sabe exatamente onde o problema está
Seleção por relevância semântica (RAG) Busca vetorial encontra trechos relevantes automaticamente Bases de código grandes, documentação extensa
Seleção por grafo de dependências Análise estática do código para incluir imports e dependências Entender efeitos colaterais de mudanças
Seleção por recência Priorizar arquivos modificados recentemente Bug em código novo, code review
Seleção heurística Regras como "sempre incluir o README, testes e schema" Setup inicial de contexto para qualquer tarefa

Exemplo prático em Claude Code

Bash# Claude Code faz seleção inteligente automaticamente:
# 1. Lê o CLAUDE.md do projeto (regras e contexto)
# 2. Usa ferramentas (Glob, Grep) para encontrar arquivos relevantes
# 3. Lê apenas as partes necessárias de cada arquivo
# 4. Mantém o histórico da conversa comprimido

Referência

Lewis, P. et al. (2020). "Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks". Facebook AI/UCL. arXiv:2005.11401. arxiv.org/abs/2005.11401 — Paper fundacional sobre RAG (Retrieval-Augmented Generation).

Pilar 2 — Organização do Contexto

A ordem e estrutura das informações no contexto impactam diretamente a qualidade da resposta:

Princípios de organização

  1. Informações mais importantes primeiro ou por último (evitar o "Lost in the Middle")
  2. Separação clara entre tipos de informação — use delimitadores:
XML<system_instructions>
Você é um assistente de código especializado em Python.
</system_instructions>

<project_context>
<!-- Arquitetura do projeto, convenções, stack -->
</project_context>

<relevant_code>
<!-- Arquivos de código relevantes para a tarefa -->
</relevant_code>

<user_request>
Corrija o bug na função de autenticação.
</user_request>
  1. Hierarquia de informação — do geral para o específico
  2. Metadados explícitos — nomes de arquivo, números de linha, caminhos

Pilar 3 — Compressão de Contexto

Quando há mais informação relevante do que espaço disponível:

Técnicas de compressão

Técnica Descrição Trade-off
Resumo automático LLM resume mensagens antigas da conversa Perde detalhes mas mantém essência
Truncamento inteligente Remove partes menos relevantes de arquivos longos Risco de perder contexto crucial
Chunking Divide documentos grandes em pedaços menores e seleciona os mais relevantes Pode perder visão do todo
Extração de esqueleto Extrai apenas assinaturas de funções, tipos e interfaces de arquivos grandes Perde implementação mas mantém estrutura
Deduplicação Remove informação repetida entre múltiplas fontes Pode perder nuances de diferentes contextos

Exemplo de compressão em Claude Code

Claude Code implementa compressão automática de contexto quando a conversa fica longa. O sistema:

  1. Identifica mensagens antigas que podem ser resumidas
  2. Mantém intactas as mensagens recentes e as mais relevantes
  3. Resume o restante preservando decisões-chave e contexto essencial
  4. Libera espaço para novas interações e código

Pilar 4 — Atualização Dinâmica do Contexto

O contexto não é estático — ele deve evoluir conforme a tarefa progride:

Exemplos de atualização dinâmica

  • Ferramentas de busca: Claude Code usa Grep e Glob para buscar informação sob demanda, em vez de carregar tudo antecipadamente
  • Resultados de execução: Resultados de testes, logs de erro e output de comandos são adicionados ao contexto em tempo real
  • Refinamento iterativo: A cada turno da conversa, o contexto é refinado com base no que foi aprendido

Seção 4.4

Engenharia de Contexto na Prática

Cenário: Você quer que Claude Code corrija um bug em uma API REST

Abordagem Ruim (sem engenharia de contexto)
"Corrige o bug na API"
→ Modelo não tem contexto suficiente
→ Precisa fazer muitas perguntas
→ Respostas genéricas
Abordagem Boa (com engenharia de contexto)
Contexto fornecido ao modelo:
1. CLAUDE.md com arquitetura do projeto
2. O arquivo da rota com bug
3. O modelo de dados relevante
4. O teste que está falhando (com output do erro)
5. Dependências diretas (services, repositories)

Prompt: "O teste test_create_user está falhando
com 'IntegrityError'. O erro está na rota
POST /users. Corrija mantendo a convenção
do projeto de usar repository pattern."

→ Modelo tem tudo que precisa
→ Resposta precisa e contextualizada
→ Segue as convenções do projeto
💡
Dica Prática

A diferença entre uma resposta genérica e uma resposta precisa quase sempre está na qualidade do contexto fornecido, não na qualidade do prompt. Invista tempo na engenharia de contexto antes de refinar o prompt.

Seção 4.5

Otimização de Tokens e Janela de Contexto

Dominar os quatro pilares é essencial, mas o trabalho do engenheiro de contexto não termina ali. Em produção, cada token importa: tokens custam dinheiro, aumentam a latência e degradam a qualidade quando usados em excesso. Esta seção traz um arsenal prático de técnicas para reduzir consumo sem perder qualidade de resposta.

🛈
Problemas que Otimização Resolve
  • Custos altos em produção — LLM-as-a-service cobra por token; contextos inflados escalam a conta linearmente.
  • Latência elevada — mais tokens de entrada significam TTFT (Time to First Token) maior e respostas mais lentas.
  • "Lost in the Middle" (Módulo 3) — modelos perdem informação quando o contexto fica gigante, degradando a precisão.
  • Context exhaustion — conversas longas estouram a janela e forçam truncamentos abruptos.
A Regra de Ouro

O melhor contexto não é o maior contexto — é o mínimo contexto suficiente para a tarefa. Cada token extra é ruído que compete pela atenção do modelo.

4.5.1 — Prompt Caching (Cache de Prompt)

Modelos modernos como Claude, Gemini e GPT-4o suportam cache de prefixo: partes estáticas e repetidas do prompt ficam cacheadas no servidor, e requisições subsequentes pagam apenas uma fração do preço por elas.

Como funciona

  • Você marca partes do prompt como cache_control (Anthropic) ou usa prefixos idênticos (OpenAI).
  • O servidor gera um hash do prefixo e armazena as representações KV internas.
  • A próxima requisição com o mesmo prefixo reutiliza o cache — sem recomputar.
  • Economia típica: até 90% de desconto nos tokens cacheados (Anthropic) e 50% (OpenAI).
  • TTL do cache: 5 minutos (Anthropic, padrão); 1 hora com tier estendido.

O que cachear

  • System prompts longos — instruções, persona, guidelines.
  • Arquivos de regras — CLAUDE.md, .cursorrules, documentação estática.
  • Documentação de referência — API docs, schemas, specs que raramente mudam.
  • Exemplos few-shot — demonstrações que se repetem entre chamadas.

Ordenação do prompt importa

Organize do mais estático ao mais dinâmico. Tudo antes do ponto de cache é reutilizado; tudo depois é recomputado.

1
System Prompt
Instruções gerais, persona — raramente muda
CACHEÁVEL
2
Regras do Projeto
CLAUDE.md, convenções, schemas
CACHEÁVEL
3
Documentação/Few-shots
Docs de API, exemplos
CACHEÁVEL (ponto de corte)
4
Histórico da Conversa
Mensagens anteriores
DINÂMICO
5
Mensagem Atual do Usuário
Entrada atual
DINÂMICO

Template com Anthropic API (Python SDK)

Pythonimport anthropic

client = anthropic.Anthropic()

response = client.messages.create(
    model="claude-sonnet-4-5",
    max_tokens=1024,
    system=[
        {
            "type": "text",
            "text": "Você é um assistente de código especializado em Python.",
        },
        {
            "type": "text",
            "text": documentacao_extensa_do_projeto,  # 50k tokens de docs
            "cache_control": {"type": "ephemeral"}  # ← ponto de cache
        }
    ],
    messages=[
        {"role": "user", "content": "Corrija o bug na função authenticate()"}
    ]
)

# Primeira chamada: paga 100% dos 50k tokens da documentação
# Chamadas seguintes (em até 5 min): paga apenas 10% desses 50k
print(response.usage)
# cache_creation_input_tokens: 50000  (primeira vez)
# cache_read_input_tokens:    50000  (chamadas seguintes, com desconto)
💡
Dica — Ponto de Cache

Você pode definir até 4 pontos de cache (breakpoints) por requisição. Posicione-os de forma que o máximo possível do prompt seja reutilizado. Em aplicações com agentes que fazem muitas chamadas curtas, o caching pode derrubar o custo em 70-85%.

Referência

4.5.2 — Compactação e Resumo do Histórico

Conversas longas esgotam rapidamente a janela de contexto. Claude Code (e outros agentes modernos) faz compactação automática quando o histórico cresce demais.

O mecanismo

  1. O agente monitora o consumo da janela de contexto.
  2. Ao ultrapassar um threshold (ex.: 80% da janela), o agente dispara uma chamada de summary sobre mensagens antigas.
  3. Um resumo estruturado substitui aquelas mensagens no histórico.
  4. Mensagens recentes e críticas (últimos N turns) permanecem intactas.
O que é preservado O que é descartado
Decisões-chave tomadas Detalhes de tool calls antigas (output bruto)
Arquivos tocados e caminhos Conteúdos intermediários de arquivos já processados
Erros encontrados e soluções aplicadas Logs verbosos e stack traces extensos
Estado atual da tarefa Tentativas descartadas
Requisitos do usuário (original e refinado) Diálogo coloquial de confirmação
⚠
Cuidado

A compactação tem custo: é uma chamada extra à LLM que cobra input+output. Em Claude Code, use /compact manualmente apenas quando estiver próximo do limite; deixe o auto-compact cuidar do resto.

4.5.3 — Seleção Cirúrgica de Arquivos

Um dos maiores desperdícios de tokens é carregar arquivos inteiros quando apenas uma função ou bloco é relevante.

NÃO faça
"Leia todos os arquivos do projeto
e me diga onde está o bug."

→ Lê 200 arquivos, 150k tokens
→ Modelo perde foco
→ Lost in the middle
→ Custo alto, resposta ruim
FAÇA
1. Grep "IntegrityError" em src/
   → aponta 3 arquivos

2. Read auth.py offset=45 limit=30
   → só a função relevante

3. Read models/user.py offset=12 limit=20
   → só o schema User

Total: ~800 tokens — 188× menos

Táticas em Claude Code

  • Grep primeiro, Read depois — use Grep para localizar, Read apenas para a vizinhança.
  • Read com offset e limit — para arquivos > 500 linhas, leia janelas de 50-100 linhas em torno da região de interesse.
  • Glob ao invés de ls recursivo — padrões de nome filtram antes de listar.
  • Evite leituras especulativas — cada Read custa tokens mesmo que o conteúdo não ajude.
Antes — Ler tudo
auth.py (inteiro)→8.400 tok
models.py (inteiro)→6.200 tok
repositories.py→5.900 tok
tests/*.py (3 arqs)→12.500 tok
Total: ~33.000 tokens
Depois — Cirúrgico
auth.py L45-75→420 tok
models/user.py L12-32→280 tok
test_auth.py L88-110→340 tok
(output de Grep)→180 tok
Total: ~1.220 tokens (27× menos)

4.5.4 — Delegação para Subagentes

Pesquisa exploratória consome uma quantidade enorme de tokens: buscar, ler, descartar, buscar de novo. O contexto principal fica poluído com material que não será usado na resposta final.

A solução arquitetural é delegar essas tarefas a subagentes (ver Módulo 13). Cada subagente trabalha em seu próprio contexto isolado, e devolve ao agente principal apenas um resumo conciso do que encontrou.

🛈
Padrão de Delegação
  1. Agente principal identifica tarefa exploratória (busca, análise ampla).
  2. Dispara um subagente com instruções específicas.
  3. Subagente explora, lê, filtra — gastando tokens no seu próprio contexto.
  4. Subagente retorna um resumo estruturado (300-800 tokens) ao agente principal.
  5. O contexto principal recebe só o essencial — permanece enxuto.

Candidatos ideais para delegar:

  • "Encontre todos os lugares onde usamos o padrão X"
  • "Analise esta lib de terceiro e me diga a API pública"
  • "Procure padrões similares em outras partes do monorepo"
  • "Classifique este CSV de 10k linhas e me dê um sumário"

4.5.5 — Técnicas de Redução de Tokens em Prompts

Mesmo prompts bem estruturados podem ser comprimidos sem perda de qualidade. A tabela abaixo lista técnicas concretas com economia estimada.

Técnica Exemplo Economia aprox.
Remover formatação redundante Eliminar headers decorativos, separadores ===, emojis supérfluos 5-15%
Abreviações consistentes fn vs function; req/res vs request/response 3-8%
Cortar cortesias Remover "por favor", "obrigado", "se possível" 2-5%
Tabelas ao invés de prosa "X tem Y e Z, A tem B e C..." → tabela Markdown 20-40% (dados tabulares)
YAML ao invés de JSON YAML omite aspas e vírgulas; menos tokens que JSON equivalente 10-30%
Inglês em prompts técnicos Tokenizers (BPE) são otimizados para inglês; português quebra mais 15-25%
Comprimir exemplos Em vez de 5 few-shots, use 2 representativos 30-60% (few-shot heavy)
Stripping de comentários Remover comentários não essenciais antes de enviar código 5-20% (código comentado)
⚠
Não Abreviação Demais

Abreviar o prompt a ponto de ficar ambíguo gera clarifying questions ou respostas incorretas — mais tokens gastos no round-trip. A regra é: comprima o que o modelo claramente infere; preserve o que é semanticamente necessário.

4.5.6 — Gerenciamento de Tokens de Saída

Lembre do Módulo 2: tokens de output custam 3-5× mais que tokens de input. Controlar a verborragia da resposta é dos lugares com maior ROI de otimização.

Diretrizes explícitas

  • Limite de parágrafos/linhas: "Responda em no máximo 3 parágrafos" / "Resposta em até 5 linhas".
  • Apenas o essencial: "Apenas o código, sem explicação." / "Responda apenas com o JSON, sem prosa."
  • Formato fixo: "Retorne no formato: {decisão}: {justificativa em 1 frase}".
  • Negativos explícitos: "Não repita o código original na resposta" / "Não inclua disclaimers".

Structured Output

JSON schemas (OpenAI response_format, Anthropic tools) restringem a resposta a campos pré-definidos — eliminando prosa introdutória e conclusiva típica dos modelos.

Python# Structured output via tool use (Anthropic)
tools = [{
    "name": "submit_fix",
    "description": "Submete a correção do bug",
    "input_schema": {
        "type": "object",
        "properties": {
            "file": {"type": "string"},
            "line": {"type": "integer"},
            "patch": {"type": "string"}
        },
        "required": ["file", "line", "patch"]
    }
}]
# Modelo é forçado a responder apenas com o objeto — nada de "Aqui está...".

Streaming

Ative stream=True para ver a resposta sendo gerada em tempo real. Isso permite cancelar a chamada assim que perceber que o modelo está divagando — evitando pagar pelo output completo.

4.5.7 — Monitoramento de Uso

Você não otimiza o que não mede. Estabeleça visibilidade sobre consumo de tokens desde o dia 1.

Ferramenta Escopo Recurso chave
Anthropic Console Conta Anthropic Usage por API key, modelo, data — console.anthropic.com/settings/usage
OpenAI Usage Dashboard Conta OpenAI Gráficos de tokens, custos, rate limits — platform.openai.com/usage
LangSmith Aplicações LangChain/custom Tracing completo de runs com tokens por passo
Helicone Proxy de observabilidade Intercepta chamadas OpenAI/Anthropic e gera métricas
LangFuse Open-source self-hosted Tracing, avaliações e custos agregados por usuário/sessão
Claude Code /status Sessão local Mostra tokens consumidos na sessão atual e janela restante
💡
Instrumente no SDK

Todos os SDKs retornam um objeto usage com input_tokens, output_tokens, cache_creation_input_tokens e cache_read_input_tokens. Emita esses valores para seu sistema de métricas (Prometheus, Datadog) a cada chamada — em produção isso é inegociável.

4.5.8 — Budget de Tokens por Tarefa

Estabeleça orçamentos de tokens por tipo de tarefa. Serve de baseline para detectar anomalias e guiar decisões de arquitetura.

Tarefa Simples (ex.: bug fix pontual)
Input (prompt + contexto)~5.000 tok
Output (patch + explicação curta)~2.000 tok
Tool calls estimadas3-6
Latência alvo< 15s
Custo estimado (Sonnet)~US$ 0,04
Tarefa Complexa (ex.: feature nova)
Input (prompt + múltiplos arqs)~50.000 tok
Output (código + testes + docs)~10.000 tok
Tool calls estimadas20-50
Latência alvo2-10min
Custo estimado (Sonnet)~US$ 0,45
🛈
Recomendação

Se uma tarefa simples está consumindo > 20k tokens de input, algo está errado: provavelmente seleção de arquivos excessiva ou histórico não compactado. Trate como alerta e investigue.

4.5.9 — Checklist de Otimização

✅
Antes de Enviar o Prompt
  • Os arquivos selecionados são realmente relevantes para esta tarefa específica?
  • Estou usando offset/limit em Read para arquivos grandes (> 300 linhas)?
  • O system prompt é estático e cacheável? Posiciono-o antes do conteúdo dinâmico?
  • Delego tarefas exploratórias a subagentes para preservar o contexto principal?
  • Limito o tamanho da resposta esperada com diretrizes explícitas?
  • Uso structured output (tools/JSON schema) quando a saída deve ser parseável?
  • Removi cortesias, emojis decorativos e prosa redundante do prompt?
  • Prefiro tabelas e YAML a listas-em-prosa e JSON quando possível?
  • Estou medindo tokens (console, LangSmith, Helicone) para detectar regressões?
  • O histórico está sendo compactado quando ultrapassa 70-80% da janela?
