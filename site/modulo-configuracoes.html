Configurações

Ajustes de ferramenta além de regras: settings.json, configuração de servidores MCP, permissões, variáveis de ambiente e integrações com IDEs.

Seção 9.1

Por Que uma Seção Separada para Configurações?

Ao personalizar uma ferramenta de IA, três conceitos costumam ser confundidos: Rules, Hooks e Configurações. Eles resolvem problemas diferentes, vivem em arquivos diferentes e são consumidos por partes diferentes do sistema.

📝 Rules Instruções textuais para o modelo — CLAUDE.md, AGENTS.md, .cursor/rules/
⚡ Hooks Automações event-driven — comandos shell disparados por eventos
⚙ Configurações O resto — settings.json, MCP, permissões, variáveis de ambiente, IDE
Analogia

Rules são o manual de conduta da IA (o que fazer, como escrever código, o que evitar). Hooks são os reflexos automáticos (quando algo acontece, execute este comando). Configurações são a preparação do ambiente de trabalho (que modelo usar, quais servidores MCP estão disponíveis, que permissões a ferramenta tem, onde estão as chaves de API).

ℹ
Onde estão as outras peças

Rules estão detalhadas no Módulo 08 — Rules. Hooks têm página dedicada (11 — Hooks). Esta página foca exclusivamente em configurações de ferramenta.

Quando cada um é a escolha certa

Você quer... Use Exemplo
Ensinar convenções de código ao modelo Rules CLAUDE.md com regras de estilo
Rodar pnpm lint após cada edição Hooks PostToolUse hook no settings.json
Trocar o modelo padrão, liberar tool X Configurações settings.json com model e permissions
Conectar o assistente ao seu Postgres Configurações mcpServers em .claude/mcp.json
Definir chave de API sem expor no repo Configurações Variáveis de ambiente e .env
Seção 9.2

settings.json (Claude Code)

O arquivo settings.json é o mecanismo central de configuração do Claude Code. Ele controla permissões, modelo padrão, hooks, servidores MCP, variáveis de ambiente e muito mais. Diferente do CLAUDE.md (que é conteúdo para o modelo), este arquivo é consumido pela própria ferramenta antes de qualquer token ser gasto.

Hierarquia de carregamento

⚙ ~/.claude/settings.json Global — aplica-se a todos os projetos do usuário
⚙ .claude/settings.json Projeto — compartilhado com o time (commitado no git)
⚙ .claude/settings.local.json Local — preferências pessoais (vai no .gitignore)
ℹ
Merge, não substituição

Os três níveis são mesclados: global → projeto → local. O nível mais específico sobrescreve o mais genérico para chaves conflitantes, mas estende listas (permissões, hooks) quando possível.

Chaves mais comuns

Chave Função Exemplo
permissions Regras granulares de quais ferramentas/comandos a IA pode usar {"allow": ["Bash(pnpm test:*)"]}
hooks Event handlers (ver Hooks) PostToolUse, UserPromptSubmit...
mcpServers Servidores MCP disponibilizados ao agente (ver MCP) {"github": {...}}
model Modelo padrão usado em novas sessões "claude-opus-4-5"
env Variáveis de ambiente injetadas no processo {"NODE_ENV": "development"}
includeCoAuthoredBy Liga/desliga linha Co-Authored-By: Claude em commits true / false
cleanupPeriodDays Dias até expiração de histórico local 30

Template completo (copy-paste)

Arquivo: .claude/settings.json

json{
  "model": "claude-opus-4-5",
  "includeCoAuthoredBy": true,
  "cleanupPeriodDays": 30,

  "env": {
    "NODE_ENV": "development",
    "PYTHONUNBUFFERED": "1"
  },

  "permissions": {
    "defaultMode": "default",
    "allow": [
      "Read(**)",
      "Edit(src/**)",
      "Edit(tests/**)",
      "Bash(pnpm test:*)",
      "Bash(pnpm lint:*)",
      "Bash(git status)",
      "Bash(git diff:*)",
      "Bash(git log:*)"
    ],
    "deny": [
      "Edit(.env)",
      "Edit(.env.*)",
      "Edit(**/secrets/**)",
      "Bash(git push:*)",
      "Bash(rm -rf:*)",
      "Bash(curl:*)"
    ]
  },

  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          { "type": "command", "command": "pnpm lint --fix $CLAUDE_FILE_PATHS" }
        ]
      }
    ]
  },

  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": { "GITHUB_TOKEN": "${GITHUB_TOKEN}" }
    }
  }
}

Seção 9.3

Modos de Permissão

O Claude Code opera em um de quatro modos de permissão, cada um com um perfil de risco diferente. O modo determina o comportamento padrão do agente; regras granulares em permissions.allow / deny continuam valendo por cima.

Modo Comportamento Quando usar Risco
default Pede confirmação humana para cada ação sensível (edição, shell, rede) Trabalho em produção, código crítico, pair programming Baixo
acceptEdits Aceita automaticamente edições de arquivos; ainda pede para shell/rede Tarefas de refatoração em massa, geração de boilerplate Médio
plan Modo só-leitura — o agente planeja, mas não executa mudanças Investigação, auditoria, exploração de código novo Muito baixo
bypassPermissions Executa qualquer ação sem prompts (YOLO mode) Sandbox descartável, container efêmero, CI isolado Alto
⚠
Cuidado com bypassPermissions

Este modo permite que o agente rode qualquer comando sem sua aprovação — incluindo rm -rf, git push --force, envio de requests autenticados, etc. Só use em ambientes descartáveis (container, VM efêmera, máquina de teste). Nunca ative em sua workstation principal.

Como configurar

Via CLI (sobrescreve o padrão apenas para a sessão):

bash# Iniciar em modo plano
claude --permission-mode plan

# Iniciar aceitando edições automaticamente
claude --permission-mode acceptEdits

# YOLO (não faça isso em casa)
claude --permission-mode bypassPermissions

Via settings.json (padrão persistente do projeto):

json{
  "permissions": {
    "defaultMode": "acceptEdits"
  }
}

Regras granulares com allow/deny

Independentemente do modo, você pode definir uma allowlist e uma denylist. A sintaxe suporta globs em caminhos e prefixos em comandos de shell:

json{
  "permissions": {
    "allow": [
      "Read(**)",
      "Edit(src/**)",
      "Bash(pnpm:*)",
      "Bash(git status)",
      "Bash(git diff:*)"
    ],
    "deny": [
      "Edit(.env)",
      "Edit(**/secrets/**)",
      "Bash(git push:*)",
      "Bash(sudo:*)"
    ]
  }
}
💡
Dica

Comece conservador (default + allowlist mínima) e afrouxe conforme observa que ações são seguras. O inverso — começar liberal e apertar depois — tende a ser muito mais doloroso.

Seção 9.4

Configuração de Servidores MCP

O Model Context Protocol (MCP) permite expor ferramentas externas (bancos de dados, APIs, filesystems remotos) ao agente. Do ponto de vista de configuração, o que importa é declarar quais servidores MCP estarão disponíveis — os detalhes do protocolo e como escrever servidores estão no Módulo 12.

Onde declarar

Servidores MCP podem ser declarados em dois lugares:

⚙ .claude/mcp.json Arquivo dedicado — recomendado para configs longas
⚙ .claude/settings.json (chave mcpServers) Dentro do settings.json principal — bom para poucos servidores

Exemplo completo (.claude/mcp.json)

json{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_TOKEN": "${GITHUB_TOKEN}"
      }
    },

    "postgres": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-postgres",
        "postgresql://localhost:5432/meu_projeto"
      ],
      "env": {
        "PGUSER": "${PGUSER}",
        "PGPASSWORD": "${PGPASSWORD}"
      }
    },

    "filesystem": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-filesystem",
        "/Users/eu/docs",
        "/Users/eu/notas"
      ]
    }
  }
}

Anatomia de um servidor MCP na configuração

Campo Descrição
command Executável que inicia o servidor (npx, uvx, docker, binário...)
args Argumentos passados ao comando — pode incluir pacote, caminhos e flags
env Variáveis de ambiente injetadas no processo (chaves, URLs, tokens)
ℹ
Interpolação de variáveis

Use ${NOME_DA_VAR} para injetar variáveis de ambiente sem precisar escrever o valor literal no arquivo. Isso permite commitar a configuração sem vazar segredos.

📚
Referência

Model Context Protocol. "Introduction". Disponível em: https://modelcontextprotocol.io

Seção 9.5

Variáveis de Ambiente

Variáveis de ambiente são o mecanismo padrão para fornecer segredos (chaves de API, tokens) e configurações sensíveis ao ambiente (URL do banco, host da API) para ferramentas de IA. Elas nunca devem ser commitadas.

Variáveis comuns

Variável Usada por Função
ANTHROPIC_API_KEY Claude Code, SDK Anthropic Autenticação com a API da Anthropic
OPENAI_API_KEY Codex CLI, SDK OpenAI, Copilot (alguns fluxos) Autenticação com a API da OpenAI
GITHUB_TOKEN MCP GitHub, gh CLI, Copilot Token de acesso pessoal ou de app do GitHub
CLAUDE_CODE_USE_BEDROCK Claude Code Roteia chamadas via AWS Bedrock
CLAUDE_CODE_USE_VERTEX Claude Code Roteia chamadas via Google Vertex AI
CLAUDE_CODE_MAX_OUTPUT_TOKENS Claude Code Limite máximo de tokens de saída por resposta
HTTP_PROXY / HTTPS_PROXY Genérica Proxy corporativo para chamadas de rede

Onde definir (em ordem de preferência)

  1. Gerenciador de segredos do SO — Keychain (macOS), Credential Manager (Windows), secret-tool (Linux).
  2. .env do projeto — carregado por direnv, dotenv, ou plugin do shell. SEMPRE no .gitignore.
  3. Shell profile (~/.zshrc, ~/.bashrc) — para variáveis globais do usuário.
  4. Cofre de CI/CD — GitHub Actions Secrets, GitLab Variables, etc., para pipelines.

Exemplo de .env de projeto

bash# .env (NUNCA commitar)
ANTHROPIC_API_KEY=sk-ant-...
GITHUB_TOKEN=ghp_...
PGUSER=eder
PGPASSWORD=supersecret
DATABASE_URL=postgresql://localhost:5432/app_dev

E o .gitignore correspondente:

gitignore# Segredos
.env
.env.*
!.env.example

# Config local de IA
.claude/settings.local.json
⚠
Nunca commite chaves de API

Chaves de API commitadas em repositórios — mesmo privados — são um vetor de ataque clássico. Bots varrem o GitHub em tempo real procurando por padrões como sk-ant-, ghp_ e AKIA. Se você vazar uma chave, revogue imediatamente e gere uma nova, mesmo que tenha removido o commit.

💡
Boas práticas
  • Mantenha um .env.example commitado com as chaves esperadas, sem valores reais.
  • Use ferramentas como GitGuardian ou git-secrets para scannear commits antes do push.
  • Rotacione chaves periodicamente, especialmente após trocas de membros do time.
  • Prefira tokens de curto prazo (OAuth) a chaves de API permanentes quando possível.
Seção 9.6

Configurações por Ferramenta

Cada ferramenta de IA tem seu próprio sistema de configuração. A tabela abaixo resume os caminhos mais importantes; lembre-se que rules (instruções de texto para o modelo) são tratadas em um arquivo diferente dentro da mesma ferramenta — isto aqui é o lado configuração.

Claude Code

⚙ ~/.claude/ Config global (settings.json, agents/, commands/)
⚙ .claude/ Config do projeto (settings.json, settings.local.json, mcp.json)
⚙ VS Code: Settings → Claude Code Integração com a extensão (mesmo schema do settings.json)

GitHub Copilot

⚙ VS Code: settings.json (github.copilot.*) Habilitar/desabilitar por linguagem, toggles de Copilot Chat
⚙ GitHub → Org Settings → Copilot Políticas organizacionais, exclusões de conteúdo
📝 .github/copilot-instructions.md Rules, não config — instruções textuais ao modelo

Cursor

⚙ ~/.cursor/ Config global do aplicativo
⚙ .cursor/ Config do projeto (incluindo regras em .cursor/rules/)
⚙ Cursor menu → Settings UI para modelo, rules, privacidade, MCP

OpenAI Codex CLI

⚙ ~/.codex/config.toml Config TOML — modelo, sandbox, approval mode, MCP
📝 AGENTS.md Rules, não config — instruções textuais ao modelo

Exemplo de ~/.codex/config.toml:

tomlmodel = "o4-mini"
approval_policy = "on-failure"
sandbox_mode = "workspace-write"

[mcp_servers.github]
command = "npx"
args = ["-y", "@modelcontextprotocol/server-github"]
env = { GITHUB_TOKEN = "${GITHUB_TOKEN}" }

Windsurf

⚙ ~/.windsurf/ Config global do aplicativo
⚙ .windsurf/ Config do projeto (workflows, rules, MCP)

Tabela comparativa rápida

Ferramenta Config global Config do projeto Formato
Claude Code ~/.claude/settings.json .claude/settings.json JSON
GitHub Copilot VS Code user settings VS Code workspace settings JSON
Cursor ~/.cursor/ + UI .cursor/ JSON / MDC
Codex CLI ~/.codex/config.toml — TOML
Windsurf ~/.windsurf/ .windsurf/ JSON / YAML
Seção 9.7

Integrações com IDEs

O surface da ferramenta — CLI, extensão de IDE, app standalone — influencia quais configurações você precisa ajustar e onde. Um agente pode existir em múltiplos surfaces simultaneamente (ex: Claude Code no terminal e na extensão do VS Code compartilham o mesmo settings.json).

Ferramenta Surfaces disponíveis Observações
Claude Code CLI, extensão VS Code, plugin JetBrains Todos compartilham ~/.claude/ e .claude/ do projeto
Cursor IDE standalone (fork do VS Code) Toda a experiência dentro do próprio app; instala extensões do VS Code marketplace
GitHub Copilot VS Code, JetBrains, Neovim, Xcode, Visual Studio, CLI Cada IDE tem extensão dedicada; config fica nas preferências do IDE
OpenAI Codex CLI + app para macOS Sem extensão oficial de IDE — executa como processo separado
Windsurf IDE standalone (Codeium) Similar ao Cursor em modelo de distribuição
Continue.dev Extensão VS Code e JetBrains Open source; config em ~/.continue/config.json
💡
Escolha do surface

Para tarefas longas e agênticas (refatoração em grande escala, criação de feature inteira), CLI ou app standalone tendem a funcionar melhor. Para completions inline e chat pontual enquanto você escreve código, extensões de IDE são insuperáveis. Muitas equipes usam os dois em paralelo.

Seção 9.8

Checklist de Configuração Inicial

Um novo projeto (ou um projeto existente adotando IA pela primeira vez) precisa passar por uma série de decisões de configuração. Use o checklist abaixo como ponto de partida.

💡
Checklist de setup inicial
  • [ ] Chaves de API em variáveis de ambiente, nunca literais em código ou config commitado.
  • [ ] settings.json do projeto commitado com as configurações que o time todo precisa (modelo, permissões base, hooks de lint).
  • [ ] settings.local.json no .gitignore para preferências pessoais não compartilhadas.
  • [ ] Permission mode apropriado definido (default em produção; acceptEdits só em contexto bem delimitado).
  • [ ] Allowlist / denylist explícita para ações críticas (proibir git push, rm -rf, edição de .env).
  • [ ] Servidores MCP necessários configurados (GitHub, banco de dados, filesystem) com variáveis interpoladas.
  • [ ] CLAUDE.md / AGENTS.md presente na raiz (ver Rules) — rules e config andam juntos.
  • [ ] Hooks relevantes (lint, teste rápido, bloqueio de secrets) configurados (ver Hooks).
  • [ ] Extensões de IDE instaladas e logadas com a mesma conta do CLI.
  • [ ] .env.example commitado com a lista de variáveis esperadas (sem valores).
  • [ ] Team alinhado: todos usando as mesmas configs base (compartilhadas via .claude/ no git).
  • [ ] Scanner de segredos (git-secrets, GitGuardian) ativo no pipeline.
  • [ ] Documentação de onboarding explicando como configurar a ferramenta localmente (chaves, MCP, permissões).
ℹ
Config vs. Rules vs. Hooks — revisão final

Se ainda houver dúvida sobre onde algo deve ir: pergunte "quem consome isso?". Se é a ferramenta (antes de qualquer token), é configuração. Se é o modelo (como contexto), é rule. Se é o sistema operacional (como comando disparado por evento), é hook.
